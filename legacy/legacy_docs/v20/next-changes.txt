This guide provides a seamless, "one-click" solution to launch remote gaming sessions on a Bazzite-Deck host from any KDE Plasma client. It automates VPN connectivity, handles session conflicts (force-closing active sessions), launches Moonlight, and cleans up the server state upon exit.

Part 1: Server-Side Setup (Bazzite-Deck)

Bazzite is an immutable OS; we install the Webhook binary natively to ensure it survives updates and has direct access to the host session management scripts.

1. Install the Webhook Binary

Download and move the standalone binary to your local bin directory:

code
Bash

sudo chmod +x /usr/local/bin/webhook
sudo chown root:root /usr/local/bin/webhook
sudo restorecon -Rv /usr/local/bin/webhook

Part 2: Client-Side Setup (KDE Desktop)
1. Import and Configure the VPN

First, import your sotohome.conf file into Network Manager:

code
Bash
download
content_copy
expand_less
nmcli connection import type wireguard file /path/to/sotohome.conf

To ensure your internet remains active while the VPN is connected and Moonlight can find the server:

Open System Settings > Connections.

Select the sotohome VPN.

Go to the IPv4 tab > Routes...

Check "IPv4 is required for this connection".

Check "Use only for resources on this connection". (This prevents the VPN from hijacking your entire internet connection).

Click Add and enter your server's LAN IP (e.g., 192.168.178.139) with Netmask 32 and Gateway 0.0.0.0.

2. Create the Master Launch Script

Create the script that orchestrates the entire session.

Paste the final script:

#!/bin/bash

# Configuration
SERVER_IP="192.168.178.139"
WEBHOOK_PORT="9000"

# Connect to VPN
if nmcli connection up sotohome; then
    sleep 3
    kdialog --passivepopup "Checking server status..." 3
    
    # Check server status and capture the response
    RESPONSE=$(curl -s -m 15 "http://$SERVER_IP:$WEBHOOK_PORT/hooks/start-gaming")

    # Check if a session is already active
    if echo "$RESPONSE" | grep -iq "ERROR: A graphical session is already active"; then
        kdialog --title "SotoHome: Session Conflict" --yesno "Do you want to stop the active session to start a new one?"
        
        if [ $? -eq 0 ]; then
            # User clicked "Yes" - Restart the session
            kdialog --passivepopup "Force-closing existing session... please wait." 5
            curl -s -m 15 "http://$SERVER_IP:$WEBHOOK_PORT/hooks/stop-session" > /dev/null
            sleep 8
            kdialog --passivepopup "Restarting gaming session..." 3
            curl -s -m 15 "http://$SERVER_IP:$WEBHOOK_PORT/hooks/start-gaming" > /dev/null
            sleep 6
        fi
        # If user clicked "No", the script just skips the restart and joins the active session
    fi

    kdialog --passivepopup "Sotohome: Ready. Launching Moonlight..." 5
    
    # Launch Moonlight and capture its exit code immediately after it closes
    if command -v flatpak >/dev/null 2>&1 && flatpak list | grep -q com.moonlight_stream.Moonlight; then
        flatpak run com.moonlight_stream.Moonlight stream hs "sotohome"
        MOONLIGHT_EXIT=$?
    else
        moonlight stream hs "sotohome"
        MOONLIGHT_EXIT=$?
    fi

    # Check if Moonlight closed normally (0) or crashed/dropped connection (non-zero)
    if [ $MOONLIGHT_EXIT -eq 0 ]; then
        # Normal exit: Run the stop-session script to clean up the host
        kdialog --passivepopup "Sotohome: Cleaning up server session..." 5
        curl -s -m 15 "http://$SERVER_IP:$WEBHOOK_PORT/hooks/stop-session" > /dev/null
    else
        # Crash/Disconnect exit: Do NOT stop the session on the host
        kdialog --passivepopup "Sotohome: Connection lost. Keeping host session alive..." 4
    fi

    # ALWAYS disconnect the VPN as a fail-safe
    sleep 2
    nmcli connection down sotohome

else
    # VPN failed to connect
    kdialog --error "Failed to connect to the Sotohome VPN."
fi

Make it executable: chmod +x ~/.local/bin/play-sotohome.sh

THRID CHANGE:

This is a formal report detailing a critical configuration conflict between Steam’s Chromium Embedded Framework (CEF) Debugger and the Nextcloud AIO Mastercontainer as configured in the current guide.

This change is required to ensure that Decky Loader and other Steam UI modifications can function alongside the headless server services.

Technical Report: Port Collision & Resolution

Subject: Resolving Port 8080 Conflict between Steam CEF and Nextcloud AIO
Status: Recommended Update for Phase 7 and Phase 10
Impact: Fixes Decky Loader/Plugin UI missing in Gaming Mode

1. Problem Identification

In the current "Bazzite Unified" configuration, Phase 7, Step 3 instructs the user to bind the Nextcloud AIO mastercontainer admin panel to 127.0.0.1:8080.

However, Steam's Gaming Mode uses a Chromium-based UI that opens a remote debugging port on localhost:8080 by default. When the Nextcloud container is running, it "hijacks" this port. As a result, tools like Decky Loader—which rely on injecting code via that debugging port—fail to connect, hitting the Nextcloud Apache server instead of the Steam UI.

Evidence from System Logs:
code
Text
download
content_copy
expand_less
Apr 02 15:39:52 hs PluginLoader[13429]: Reason: You're speaking plain HTTP to an SSL-enabled server port.
Apr 02 15:39:52 hs PluginLoader[13429]: <address>Apache/2.4.66 (Unix) Server at localhost Port 8080</address>

The logs confirm that Decky Loader is attempting to reach Steam but is being answered by the Nextcloud (Apache) container.

2. Proposed Solution: Port Reassignment to 9443

To resolve this conflict, the Nextcloud AIO admin port must be moved. Port 9443 is the optimal choice for the following reasons:

Symmetry: The guide already uses port 9443 for the SSH tunnel in Phase 10. Matching the internal server port to the external tunnel port reduces cognitive load.

Stability: Port 9443 is a standard alternative for HTTPS/SSL services and does not conflict with any other Bazzite, Sunshine, or Docker services used in this guide.

Security: By maintaining the 127.0.0.1: prefix, the port remains bound to the internal loopback and is invisible to the outside world, preserving the "Zero New Open Ports" design philosophy.

3. Implementation Steps (Guide Updates)
Update A: Phase 7, Step 3 (Nextcloud All-in-One)

The docker-compose.yml for Nextcloud should be updated to re-route the host port.

4. Final Validation

After applying these changes, the following state is achieved:

Steam CEF Debugger: Regains control of port 8080. Decky Loader successfully injects the plugin menu icon into the Steam sidebar.

Nextcloud AIO: Operates securely on port 9443.

User Workflow: The user still visits https://localhost:9443 in their browser to manage Nextcloud, maintaining consistency with existing documentation.

Conclusion: This change is essential for a "Unified" machine where both server background tasks and gaming frontend features must coexist without interference.
