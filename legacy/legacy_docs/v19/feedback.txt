ISSUE 1 — CRITICAL: stop-session.sh Destroys Sunshine's Encoder State
--------------------------------------------------------------------------------

PHASE AFFECTED: Phase 8 (Session Management Scripts)
SEVERITY: Critical — renders Sunshine permanently broken until manual recovery,
          survives reboots, requires a user session to be active for recovery.

PROBLEM DESCRIPTION:
The stop-session.sh script as written in the guide causes a hard, persistent
failure in Sunshine's hardware encoder that survives reboots. After running
stop-session.sh, connecting via Moonlight produces an "Error code 503" and
Sunshine reports it cannot find a video encoder and to ensure a screen is
connected. This state does not self-heal. It persists after reboots.

ROOT CAUSE — TWO COMPOUNDING BUGS:

Bug 1: Hard session termination via loginctl terminate-session.
The guide's stop-session.sh uses loginctl terminate-session to end the graphical
session. This is a SIGTERM shotgun blast — it kills all processes in the session
cgroup simultaneously without any compositor shutdown sequence. The GPU/DRM
framebuffer and VA-API encoder context are yanked out from under Sunshine
mid-operation. The DRM master lock is not released cleanly. This is the
equivalent of pulling the power on the compositor while Sunshine has the GPU in
active use.

Bug 2: Sunshine is restarted without an active display immediately after.
After terminating the session, stop-session.sh restarts SDDM and then restarts
Sunshine. At the moment Sunshine restarts, the SDDM greeter is in the process of
starting but the display compositor is not yet fully live. Sunshine performs
encoder detection at startup — this is a one-time probe, not a continuous
negotiation. If Sunshine starts and finds no valid Wayland display or an
incomplete one, it commits to a degraded/failed encoder state permanently until
manually reset. It cannot re-probe on its own.

The start scripts (start-kde.sh, start-gaming.sh) contain detailed comments
explaining exactly this failure mode and why they restart Sunshine only AFTER
the user session is confirmed live with WAYLAND_DISPLAY propagated. The
stop-session.sh script was written without applying the same reasoning, creating
an exact instance of the failure the start scripts were designed to avoid.

The irony is that the fix for the start scripts was already in the codebase —
stop-session.sh simply needed to apply the same logic in reverse.

WHY THE KDE LOGOUT BUTTON WORKS BUT stop-session.sh DOES NOT:
This is the key diagnostic observation. A user who logs out of KDE using the
Application Menu logout button can immediately reconnect via Moonlight to the
SDDM greeter — Sunshine keeps streaming without interruption. The stop-session.sh
script causes a fatal crash. The difference is:

  KDE logout button:
    - KWin compositor performs graceful shutdown sequence
    - Wayland clients (including Sunshine) receive protocol-level disconnect
    - DRM master is released cleanly before SDDM takes over
    - Sunshine keeps running, keeps its encoder context, streams greeter

  stop-session.sh (original):
    - loginctl terminate-session kills everything simultaneously
    - Sunshine, KWin, and all other processes die together
    - DRM state is dirty
    - stop-session.sh then immediately restarts Sunshine into a display void
    - Sunshine commits to broken encoder state

ADDITIONAL SYMPTOM: stop-session.sh kills the user's SSH connection.
If the user runs stop-session.sh from a terminal that is running inside the
Moonlight/Sunshine stream (i.e., a terminal emulator in KDE Plasma being
streamed), killing the KDE session kills the terminal, which kills the SSH
connection. This is expected but undocumented and confusing. The guide should
explicitly state: always run stop-session.sh from a native SSH connection on
port 22022, never from a terminal inside a Moonlight stream.

RECOVERY PROCEDURE (for users who have hit this state):
  systemctl --user reset-failed sunshine
  systemctl --user restart sunshine  # Only works if a display session is active
  # If still broken: run sudo start-kde.sh first, then let the script restart
  # Sunshine at the end of its execution with a live WAYLAND_DISPLAY.

THE FIX — Graceful logout, no Sunshine restart:
stop-session.sh must be rewritten with two principles:

  Principle 1: Use graceful, session-type-aware logout instead of hard kill.
    For KDE: send the logout dbus signal (org.kde.Shutdown.logout) — this is
    exactly what the KDE logout button does. KWin shuts down cleanly.
    For Gaming Mode: stop gamescope-session@steam.service via systemctl --user.
    Hard-kill (loginctl terminate-session) is retained only as a fallback after
    a 15-second graceful timeout.

  Principle 2: Never restart Sunshine from stop-session.sh under any
  circumstances. After a graceful logout, Sunshine keeps its encoder context
  and continues streaming the SDDM greeter. There is nothing to fix.
  Restarting it here is what breaks it.

REVISED stop-session.sh LOGIC:
  1. Cancel stale restore-headless timers (same as before)
  2. Write headless SDDM config immediately (same as before)
  3. Detect active seat0 user session (same as before)
  4. Detect session type: check if gamescope-session@steam.service is active
     to distinguish Gaming Mode from KDE
  5. Attempt graceful logout appropriate to session type
  6. Wait up to 15s for clean exit
  7. Hard-kill fallback only if timeout exceeded
  8. Poll for SDDM greeter (no SDDM restart — greeter appears on its own
     after clean session exit)
  9. Last-resort SDDM restart only if greeter never appears after 20s
  10. Sunshine is NOT touched — add explicit log comment stating this

IMPORTANT NOTE ON SDDM RESTART REMOVAL:
The original stop-session.sh restarted SDDM after session termination. This is
unnecessary after a graceful logout — SDDM automatically reclaims the display
and shows the greeter when the user session exits. Forcing a restart adds
unnecessary delay and another potential display context disruption. It should
only happen as a last resort if the greeter does not appear on its own.

ISSUE 2 — IMPORTANT: Missing User Group Memberships
--------------------------------------------------------------------------------

PHASE AFFECTED: Phase 9, Step 1b (Host OS Preparation)
SEVERITY: Important — causes Sunshine to fail, silent and confusing

PROBLEM DESCRIPTION:
Phase 9 Step 1b instructs the user to add themselves to the input, video, and
render groups using:

  sudo usermod -aG input,video,render $(whoami)

This is correct as written BUT the guide only documents adding to the docker
group in Phase 5, and the connection between group membership and Sunshine's
specific capabilities is never explained. In practice, a user following the guide
may already have partial group membership from earlier steps or from Bazzite
defaults, and not realise which groups are actually required for what.

The confirmed working group set for the sotohome user on this deployment is:

  sotohome wheel video input render docker

These groups provide:
  wheel   — sudo access (standard)
  video   — access to /dev/dri/card0 (display output, DRM master operations,
             KMS capture by Sunshine in kms mode)
  input   — access to /dev/input/* (keyboard and mouse capture for streaming)
  render  — access to /dev/dri/renderD128 (VA-API hardware encoding via GPU)
  docker  — Docker daemon socket access (Phase 5)

The guide should explicitly list all five groups in a single consolidated place,
with a clear explanation of why each one is needed for Sunshine specifically.
Phase 9 Step 1b is the right location.

Additionally: the guide correctly notes in Pitfall #16 that group changes require
a full reboot. This is correct and should be reinforced at Step 1b with a
reminder that if Sunshine fails to start after the reboot, the first diagnostic
is to verify all five groups are present with: groups

If any group is missing, add it and reboot again. There is no partial fix.

ISSUE 3 — ARCHITECTURAL: Sunshine wlr Capture Requires Restart on Every
          Session Change — Root Cause and Permanent Fix
--------------------------------------------------------------------------------

PHASE AFFECTED: Phase 9 (WireGuard VPN + Sunshine/Moonlight Streaming),
                Phase 8 (Session Management Scripts)
SEVERITY: Architectural improvement — eliminates the entire class of session
          transition failures. Highly recommended for all deployments.

BACKGROUND — WHY THE SCRIPTS RESTART SUNSHINE:
Both start-kde.sh and start-gaming.sh contain a Sunshine restart at the end.
The comments in the scripts explain the reason: Sunshine must be restarted after
each new session starts so it can re-probe WAYLAND_DISPLAY and acquire the
correct display context. Without this restart, Sunshine remains pointed at a
dead or stale display socket from a previous session.

This requirement exists because Sunshine's default capture mode on Wayland is
"wlr" (wlroots Wayland capture protocol). wlr capture works by connecting to
a specific WAYLAND_DISPLAY socket at startup. That socket is owned by the
Wayland compositor running at the time:

  SDDM greeter: WAYLAND_DISPLAY=wayland-0 (owned by the sddm system user)
  KDE Plasma:   WAYLAND_DISPLAY=wayland-1 (owned by the login user, KWin)
  Gamescope:    WAYLAND_DISPLAY=wayland-0 or wayland-1 (gamescope compositor)

When the session changes, the socket changes. Sunshine does not dynamically
switch — it holds the old dead socket. A restart is required to re-probe.

This means every session transition in the setup is a potential failure point:
  - If Sunshine is restarted too early (display not ready): broken encoder state
  - If Sunshine is restarted too late (session already changed): missed context
  - If the restart itself fails silently: Sunshine appears active but cannot stream

The start scripts handle this with careful polling (wait for seat0 session,
buffer for compositor init, then restart). This works but is fragile and slow
(adds 5-15 seconds to every session start).

THE ROOT FIX — Switch Sunshine to KMS Capture Mode:

The kms (Direct KMS/DRM) capture mode reads the GPU framebuffer directly at the
kernel level via /dev/dri/card0. It does not connect to a Wayland socket. It
does not care who owns the compositor. It captures whatever is physically
displayed on the output connected to the GPU — in this setup, the HDMI dummy
plug — regardless of what session, compositor, or display manager is running.

With kms capture:
  - Sunshine streams the SDDM greeter from the first boot, no session required
  - When KDE starts, Sunshine streams KDE automatically — no restart
  - When KDE exits, Sunshine streams the greeter automatically — no restart
  - When Gaming Mode starts, Sunshine streams gamescope automatically — no restart
  - When Gaming Mode exits, Sunshine streams the greeter automatically — no restart
  - stop-session.sh becomes trivially simple — just exit the session gracefully,
    Sunshine handles the rest
  - The entire polling-and-restart block in start-kde.sh and start-gaming.sh
    can be removed entirely

PREREQUISITE — HDMI/DP Dummy Plug (already mandatory per guide Pitfall #15):
kms capture requires an active display output to read from. The guide already
mandates a dummy plug for headless GPU initialisation (Pitfall #15). Since the
dummy plug is already required, kms capture has no additional hardware
requirements. Users following this guide already have everything needed.

PREREQUISITE — Group membership:
kms capture requires the video group for /dev/dri/card0 access. The video group
is already required and documented in Phase 9 Step 1b. No additional groups needed.

HOW TO CONFIGURE KMS CAPTURE:

Method A — Sunshine web UI (recommended):
  1. Open SSH tunnel: ssh -N -L 47990:127.0.0.1:47990 -p 22022 username@server-ip
  2. Go to https://localhost:47990
  3. Configuration → Video → Capture → select "kms"
  4. Save → restart Sunshine: systemctl --user restart sunshine

Method B — Config file:
  grep -q '^capture' ~/.config/sunshine/sunshine.conf \
    && sed -i 's/^capture.*/capture = kms/' ~/.config/sunshine/sunshine.conf \
    || echo 'capture = kms' >> ~/.config/sunshine/sunshine.conf
  systemctl --user restart sunshine

VERIFICATION:
After switching to kms capture and restarting Sunshine, verify from a known-good
headless state (no user session active, only SDDM greeter):
  1. loginctl list-sessions  # confirm: greeter only, no user session on seat0
  2. systemctl --user is-active sunshine  # confirm: active
  3. Connect Moonlight to 10.8.0.1
  4. Expected: SDDM login screen is visible and streaming in Moonlight
If this works, kms capture is functioning correctly. All subsequent session
transitions will be seamless without any script intervention.

GAMESCOPE-SPECIFIC NOTE:
Gamescope can use direct GPU scanout (bypassing the compositor layer entirely).
In direct scanout mode, it writes to the DRM plane directly. kms capture reads
at the DRM framebuffer level and sees this correctly. However, this should be
explicitly tested by the user after switching to kms capture:
  sudo start-gaming.sh  # launch Gaming Mode
  Connect Moonlight → verify Steam Big Picture is visible
If it is, kms capture works end-to-end for all three display states (greeter,
KDE, Gaming Mode).

IMPACT ON START SCRIPTS:
With kms capture confirmed working, the Sunshine restart blocks at the end of
start-kde.sh and start-gaming.sh should be removed. These blocks were
compensating for the limitation of wlr capture. They are not just unnecessary
with kms — they add 5-20 seconds of blocking delay to every session start for
no benefit.

The simplified start-kde.sh ends at step 5 (schedule headless restoration).
The simplified start-gaming.sh ends at step 6 (schedule headless restoration).
No polling, no Sunshine restart, no wait loops.

IMPACT ON stop-session.sh:
With kms capture, stop-session.sh only needs to:
  1. Cancel stale timers
  2. Write headless config
  3. Detect session type
  4. Graceful logout (KDE dbus or gamescope service stop)
  5. Hard-kill fallback if graceful timeout exceeded
No Sunshine restart. No SDDM restart. No polling for greeter.
Sunshine streams the greeter the moment the session exits.

SUMMARY OF ARCHITECTURAL CHANGE:
The wlr capture model treats Sunshine as a client of whichever compositor is
currently running, requiring orchestration (restart + timing) on every session
change. The kms capture model treats Sunshine as an independent observer of the
display framebuffer, requiring no orchestration at all. For this setup, where
the fundamental design goal is seamless remote access across all system states,
kms capture is the correct architectural choice.

The only scenario where wlr capture would be preferable is if the user needs
Wayland-specific features like per-application window capture or virtual display
creation. For full-screen desktop and Gaming Mode streaming — which is all this
guide covers — kms is strictly superior.

ISSUE 4 — MINOR: ujust setup-sunshine May Require Manual Group Additions
--------------------------------------------------------------------------------

PHASE AFFECTED: Phase 9, Step 1b
SEVERITY: Minor — silent failure, confusing to diagnose

PROBLEM DESCRIPTION:
The guide correctly instructs running sudo ujust setup-sunshine and then
sudo usermod -aG input,video,render $(whoami). In practice, ujust setup-sunshine
may not add all required groups on all Bazzite builds, or the groups may not
apply correctly until after the reboot.

The confirmed workaround for this deployment was to add the groups manually in
the same style the guide uses for the docker group in Phase 5, not relying
solely on ujust. The guide should note that if Sunshine fails to start after
the reboot in Step 1c, the first check should always be:

  groups
  # Expected: wheel video input render docker (plus any other groups)

And if any of video, input, or render are missing:
  sudo usermod -aG video,input,render $(whoami)
  sudo systemctl reboot

This is a diagnostic note, not a change to the primary instructions.

--------------------------------------------------------------------------------

Phase 8 — Session Management Scripts:

  1. stop-session.sh: Replace entirely with graceful-logout version.
     Key changes:
       - Detect session type (KDE vs Gaming Mode) before attempting logout
       - KDE: use qdbus org.kde.Shutdown /Shutdown logout (same as logout button)
       - Gaming Mode: use systemctl --user stop gamescope-session@steam.service
       - Hard-kill (loginctl terminate-session) is fallback only, after 15s timeout
       - Remove all SDDM restart logic (last-resort only after 20s greeter timeout)
       - Remove all Sunshine restart logic entirely
       - Add explicit log comment: Sunshine is intentionally not touched
     Add warning box: always run stop-session.sh from a native SSH connection,
     never from a terminal inside a Moonlight stream.

  2. start-kde.sh and start-gaming.sh: Remove Sunshine restart blocks entirely
     AFTER confirming kms capture is working (see Phase 9 change below).
     The scripts become shorter and faster — no polling delays.
     Add a comment noting kms capture makes Sunshine restart unnecessary.

Phase 9 — WireGuard VPN + Sunshine/Moonlight Streaming:

  3. Add new step after Step 1b: "Switch Sunshine to KMS Capture Mode"
     Position: between Step 1 (Host OS Preparation) and Step 2 (wg-easy).
     This is the correct position because it requires Sunshine to be installed
     (done by ujust setup-sunshine in Step 1b) but does not require WireGuard
     to be configured yet.
     Content:
       - Explain why kms is preferred over wlr for this setup (see Issue 3)
       - Explain the prerequisite: dummy plug (already mandatory, no new hardware)
       - Explain the prerequisite: video group (already required)
       - Provide both configuration methods (web UI and config file)
       - Provide verification procedure (Moonlight → greeter while headless)
       - Note the gamescope direct-scanout test
       - Note that after this step the Sunshine restart blocks in the start
         scripts are removed (forward reference to the updated Phase 8 scripts)

  4. Step 1b group membership: Add explicit list of all five required groups
     (wheel, video, input, render, docker) with per-group explanation of what
     each enables for Sunshine. Add diagnostic note about using groups to verify.

Appendix A — Troubleshooting:

  5. Add new troubleshooting entry:
     "Moonlight shows Error 503 / Sunshine reports no video encoder after
     running stop-session.sh"
     Cause: stop-session.sh restarted Sunshine without an active display.
     Fix: systemctl --user reset-failed sunshine, then run sudo start-kde.sh
     to get a live session, then Sunshine will restart cleanly at the end of
     the script. Long-term fix: switch to kms capture (Phase 9 new step).

  6. Add note to existing "Sunshine not streaming / Moonlight cannot connect"
     entry: if stop-session.sh was recently run and the issue started immediately
     after, this is the encoder state bug — see entry 5 above.

  7. Add note: "stop-session.sh disconnects my terminal"
     Cause: terminal is running inside KDE being streamed via Moonlight.
     Fix: always run stop-session.sh from a native SSH connection on port 22022.

Appendix B — Normal Workflows:

  8. Update all workflow descriptions to reflect that with kms capture, Moonlight
     can connect to the server at any time — including before any start script
     is run — and will display the SDDM greeter. This is the new expected baseline
     behaviour. The user does not need to run a start script just to get a
     Moonlight connection. Start scripts are for launching a specific graphical
     environment, not for enabling Moonlight access.

  9. Add entry: "Sunshine is always streaming — connect anytime"
     With kms capture and the dummy plug, Sunshine streams whatever is on screen
     from the moment of boot. Connect Moonlight to 10.8.0.1 at any time to see
     the current display state without running any SSH command first.
