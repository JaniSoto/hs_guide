ROCM + 780M — DEFINITIVE ASSESSMENT
════════════════════════════════════════════════════════════════════

The 780M (gfx1103 / RDNA3) is usable for GPU inference on Bazzite (Fedora 42).
Three environment variables and one kernel parameter are mandatory.

Required environment variables in the Ollama docker-compose.yml:
  - HSA_OVERRIDE_GFX_VERSION=11.0.2
  - OLLAMA_FLASH_ATTENTION=false

  HSA_OVERRIDE_GFX_VERSION=11.0.2
    Bypasses ROCm's hardware allowlist. Without it, Ollama silently falls back
    to CPU. Must be in the container's environment block in docker-compose.yml —
    setting it in the host shell has no effect on containers.

  OLLAMA_FLASH_ATTENTION=false
    RDNA3 has a numerical instability bug in Flash Attention causing output to
    degenerate into repeated tokens (e.g. "444444..."). Disabling it eliminates
    the issue with negligible performance impact.

Required kernel parameter — mandatory before any deployment:
  sudo rpm-ostree kargs --append=amdgpu.noretry=0
  systemctl reboot

  Without amdgpu.noretry=0, the iGPU can suffer an unrecoverable MES
  (Micro Engine Scheduler) hang during sustained inference.

Expected performance:
  CPU-only:    ~3–6 tok/s (7B Q4)
  780M GPU:    ~15–20 tok/s (7B Q4) · ~10–14 tok/s (14B Q4)
  Effect:      Voice end-to-end latency: ~30–40s (CPU) → ~8–12s (GPU)

BIOS prerequisite:
  The 780M uses system RAM as VRAM. A 14B Q4 model needs ~9 GB.
  BIOS UMA Frame Buffer Size must be "Auto" or ≥8 GB. GMKtec default: Auto.

eGPU note:
  When an OCuLink eGPU (RX 6000/7000 series) is added, Ollama auto-selects
  it with no configuration changes. Use ROCR_VISIBLE_DEVICES if explicit
  device selection is needed.

DOCKER NETWORK TOPOLOGY
───────────────────────

  Create once before deploying any AI stack container:
    docker network create ai-net

  This is a persistent host resource. It survives compose up/down cycles.

  Network membership:
    ai-net (bridge):  ollama, openwebui, mcpo
    host network:     homeassistant
    own default:      wyoming-whisper, wyoming-piper (ports published to host)

  Why HA is not on ai-net:
    network_mode: host bypasses all Docker bridge networks. HA cannot join
    ai-net. It reaches other services via 127.0.0.1 (loopback).

  Hostname resolution matrix:
    openwebui     → ollama:11434         via ai-net (internal Docker hostname)
    openwebui     → mcpo:8000            via ai-net (internal Docker hostname)
    homeassistant → 127.0.0.1:11434      loopback (Ollama published port)
    homeassistant → 127.0.0.1:10300      loopback (wyoming-whisper)
    homeassistant → 127.0.0.1:10200      loopback (wyoming-piper)
    NPM           → <LAN-IP>:3000        NPM is on nextcloud-aio-network;
    NPM           → <LAN-IP>:8123        cross-network hostname resolution
                                         does not work — use LAN IP.

  Why Ollama and Wyoming ports bind to 127.0.0.1:
    Default Docker binding (0.0.0.0) exposes unauthenticated endpoints to all
    LAN devices. Since HA is on host network and shares the loopback, binding
    to 127.0.0.1 gives HA full access while denying LAN-wide access.
    Open WebUI is unaffected — it uses http://ollama:11434 over ai-net.

OLLAMA (ollama/ollama:rocm)
───────────────────────────
The :rocm tag is mandatory. The untagged ollama/ollama image is CPU-only
and ignores all GPU environment variables.

  services:
    ollama:
      image: ollama/ollama:rocm
      container_name: ollama
      restart: unless-stopped
      ports:
        - "127.0.0.1:11434:11434"
      devices:
        - /dev/kfd:/dev/kfd
        - /dev/dri:/dev/dri
      group_add:
        - "<VIDEO_GID>"         # numeric GID from EX.0.d — e.g., "44"
        - "<RENDER_GID>"        # numeric GID from EX.0.d — e.g., "107"
      environment:
        - HSA_OVERRIDE_GFX_VERSION=11.0.2
        - OLLAMA_FLASH_ATTENTION=false
        - OLLAMA_KEEP_ALIVE=5m
        - OLLAMA_NUM_PARALLEL=1
        - OLLAMA_HOST=0.0.0.0
      volumes:
        - ./models:/root/.ollama
      networks:
        - ai-net

  networks:
    ai-net:
      external: true

  Why numeric GIDs in group_add:
    Docker resolves string group names from the container's own /etc/group,
    not the host's. If the group is absent from the container image, startup
    crashes with "Unable to find group render". Obtain with:
      getent group video && getent group render

  Recommended model: qwen2.5:14b-instruct-q4_K_M
    Valid Ollama tag, 21M+ downloads, 9.0 GB, ~10–14 tok/s on 780M.
    Significantly better than 7B on multi-step queries and tool use.
    Fallback: mistral:7b-instruct-q4_K_M (~5 GB) if RAM pressure is observed.

  GPU verification:
    docker exec ollama ollama ps
    → PROCESSOR must show "100% GPU" during inference.
    If CPU: confirm HSA_OVERRIDE_GFX_VERSION is present with:
      docker inspect ollama | grep -A1 HSA

MCPO (ghcr.io/open-webui/mcpo:main)
────────────────────────────────────
mcpo requires an explicit startup command. Without it the container starts
with no MCP servers loaded and exposes no tools.

  config.json:
    {
      "mcpServers": {
        "filesystem": {
          "command": "npx",
          "args": ["-y", "@modelcontextprotocol/server-filesystem",
                   "/nc-data"]
        },
        "memory": {
          "command": "npx",
          "args": ["-y", "@modelcontextprotocol/server-memory"],
          "env": {
            "MEMORY_FILE_PATH": "/app/memory/graph.json"
          }
        }
      }
    }

  MEMORY_FILE_PATH is an environment variable read by the memory server
  package, not a CLI argument. Without it, the memory graph is not persisted
  to disk and is lost on every container restart.

  The filesystem path argument "/nc-data" scopes the server to a specific
  user's files only — see the volume mount below.

  docker-compose.yml:
    services:
      mcpo:
        image: ghcr.io/open-webui/mcpo:main
        container_name: mcpo
        restart: unless-stopped
        command: --config /app/config/config.json
        user: "<NC_UID>:<NC_GID>"        # from EX.0.g — e.g., "33:33"
        environment:
          - npm_config_cache=/tmp/npm_cache
        volumes:
          - ./config.json:/app/config/config.json:ro
          - /var/srv/nextcloud/data/<your_nc_username>/files:/nc-data:ro
          - ./memory:/app/memory
        networks:
          - ai-net

    networks:
      ai-net:
        external: true

  Why /data/<username>/files (not the full /data path):
    Mounting the entire Nextcloud data directory includes system folders
    (appdata_*, updater-*, admin config) that pollute the AI's context
    with irrelevant content. Scoping to one user's files directory is
    cleaner and avoids exposing SSE-encrypted blobs if server-side
    encryption is enabled (the AI would otherwise read encrypted garbage).
    Replace <your_nc_username> with your actual Nextcloud username.

  Why npm_config_cache=/tmp/npm_cache:
    mcpo runs as a non-root UID. When npx downloads MCP server packages,
    npm tries to write its cache relative to that user's home directory,
    which may not exist or may not be writable inside the container.
    Redirecting to /tmp prevents EACCES crashes. No pre-creation needed.

  Pre-deployment requirement (CRITICAL):
    The ./memory directory must exist and be owned by NC_UID:NC_GID before
    docker compose up. Docker auto-creates missing bind-mount dirs as root:root,
    preventing mcpo from writing the memory graph on startup.

      mkdir -p ~/docker/mcpo/memory
      sudo chown <NC_UID>:<NC_GID> ~/docker/mcpo/memory

  Connecting Open WebUI to mcpo — per-server registration:
    Each MCP server is accessible at its own URL path under mcpo.
    Adding the root http://mcpo:8000 discovers no tools.
    Register each server individually:

      Admin Settings → Tools → "+" (Add Tool Server)
        URL: http://mcpo:8000/filesystem   Type: OpenAPI   Auth: None
        URL: http://mcpo:8000/memory       Type: OpenAPI   Auth: None

    Enable both tools globally in Admin Settings and per-model in model config.

  Filesystem server reads: plain text files (.txt, .md, .py, .json, .yaml,
  config files). Does NOT parse EPUB (ZIP archives return binary data). Use
  a dedicated eBook-mcp server as a third config.json entry if needed.

  Memory usage: "Remember my ID is X" → stored. "What's my ID?" → retrieved
  across sessions and restarts. "Forget my ID" → deleted.


HOME ASSISTANT (ghcr.io/home-assistant/home-assistant:stable)
──────────────────────────────────────────────────────────────
Official HA-maintained image. No PUID/PGID. Requires privileged mode and
dbus access as documented at home-assistant.io/installation/linux.

  services:
    homeassistant:
      image: ghcr.io/home-assistant/home-assistant:stable
      container_name: homeassistant
      restart: unless-stopped
      privileged: true
      network_mode: host
      environment:
        - TZ=Europe/Berlin          # TZ database format — adjust if needed
      volumes:
        - ./config:/config
        - /etc/localtime:/etc/localtime:ro
        - /run/dbus:/run/dbus:ro
      labels:
        com.centurylinklabs.watchtower.enable: "false"

  Why privileged: true:
    The official HA Docker installation documentation requires it for full
    hardware access — including Bluetooth, GPIO, and USB device passthrough.
    It is not specific to HAOS or the Supervised method; it applies to HA
    Container when hardware integrations are used.
    Future path: once a Zigbee dongle is added (EX.9), privileged: true can
    be replaced with a targeted devices: entry pointing to the dongle's
    /dev/serial/by-id/ path, which is the more secure long-term approach.
    For the core build (EX.0–EX.6) with no USB hardware, this distinction
    is moot — keep privileged: true as documented.

  network_mode: host is required for mDNS/UPnP device discovery. There must
  be no ports: block. HA add-ons do not exist. Do not follow any tutorial
  referencing the Supervisor, Add-on Store, or named add-ons.

  Reverse proxy configuration (required before remote access):
    Since HA 2021.7, reverse proxy requests are rejected by default with
    "400 Bad Request". This must be configured before testing external access.
    Edit ~/docker/homeassistant/config/configuration.yaml:

      http:
        use_x_forwarded_for: true
        trusted_proxies:
          - 172.16.0.0/12    # standard Docker bridge range (172.x.x.x)
          - 192.168.0.0/16   # guards against non-standard Docker allocations
          - 127.0.0.1        # loopback

    Why 172.16.0.0/12: NPM is a Docker container. The source IP HA sees for
    proxied requests is the NPM container's bridge IP, which falls in the
    172.x.x.x range under standard Docker allocation. 192.168.0.0/16 is
    added as a safety net for cases where Docker allocates custom networks
    outside the standard range.

    Validate the YAML before restarting (YAML is indentation-sensitive;
    a single space error will prevent HA from booting):
      docker exec homeassistant python -m homeassistant \
        --script check_config --config /config
      → must print "Configuration is valid!"

    Only restart after validation passes:
      cd ~/docker/homeassistant && docker compose restart

  NPM proxy forward target: http://<LAN-IP>:8123
    NPM cannot resolve a host-networked container by hostname across separate
    Docker networks. Always use the server's LAN IP as the forward target.

IMPLEMENTATION PHASES
════════════════════════════════════════════════════════════════════

PHASE 0 — KERNEL PARAMETER (mandatory, requires reboot)
  Execute:
    sudo rpm-ostree kargs --append=amdgpu.noretry=0
    systemctl reboot

  Verify after reboot:
    cat /proc/cmdline | grep amdgpu.noretry
    → must contain "amdgpu.noretry=0"

  Do not proceed until confirmed.

──────────────────────────────────────────────────────────────────

EX.0  Pre-flight (~20 min)
  All values gathered here feed directly into compose files. Record them.

  a. Verify existing containers healthy:
       docker ps --format "table {{.Names}}\t{{.Status}}"
       → duckdns, npm, nextcloud-aio-mastercontainer: all "Up"

  b. Confirm kernel parameter (see Phase 0).

  c. Verify GPU device nodes exist:
       ls -la /dev/kfd /dev/dri/renderD128

  d. Retrieve numeric GIDs for Ollama group_add. Record both numbers:
       getent group video    → third field is the GID (e.g., 44)
       getent group render   → third field is the GID (e.g., 107)

  e. Verify available RAM:
       free -h | awk '/^Mem:/{print $7}'
       "Available" includes reclaimable cache. Aim for ≥18 GB.

  f. Verify BIOS UMA Frame Buffer Size:
       Boot into BIOS → APU/Advanced → UMA Frame Buffer Size
       Must be "Auto" or ≥8 GB (GMKtec default: Auto).

  g. Retrieve Nextcloud data owner UID:GID for mcpo. Record both numbers:
       ls -lnd /var/srv/nextcloud/data
       The -n flag forces numeric output. The -d flag lists the directory
       itself, not its contents. Note the third (UID) and fourth (GID) fields.

  h. Identify your Nextcloud username for the mcpo volume mount path:
       ls /var/srv/nextcloud/data/
       → your username appears as a directory alongside appdata_* and others

  i. Create shared Docker network:
       docker network create ai-net

CRITICAL PITFALLS
════════════════════════════════════════════════════════════════════

1.  Kernel parameter before everything.
    rpm-ostree kargs + reboot required first. iGPU hangs the OS under
    sustained inference without it.

2.  Wrong Ollama image tag.
    ollama/ollama:rocm is the GPU image. The untagged image is CPU-only.

3.  HSA_OVERRIDE_GFX_VERSION must be in the compose environment block.
    Shell-level env vars do not propagate into containers.

4.  OLLAMA_FLASH_ATTENTION=false is mandatory on gfx1103.
    RDNA3 Flash Attention causes output to collapse into repeated tokens.

5.  Numeric GIDs in group_add, not string names.
    String names are resolved from the container's /etc/group, not the host.
    Startup crashes if the group is absent from the container image.

6.  ai-net must exist before docker compose up on any referencing stack.
    Run docker network create ai-net before all other steps.

7.  Ollama and Wyoming ports bound to 127.0.0.1, not 0.0.0.0.
    HA connects via 127.0.0.1 (loopback, host network).
    Configuring HA's Ollama URL as "http://ollama:11434" will fail — HA is
    not on ai-net and cannot resolve that hostname.

8.  mcpo requires command: --config /app/config/config.json.
    Without it the container starts with no tools loaded.

9.  MEMORY_FILE_PATH is an env var in config.json, not a CLI argument.
    Without it, the memory graph is not written to disk.

10. mcpo tools must be registered individually in Open WebUI.
    http://mcpo:8000/filesystem and http://mcpo:8000/memory — not the root.
    The root URL exposes no tool definitions.

11. mcpo memory directory must be pre-chowned before docker compose up.
    Docker auto-creates missing bind-mount dirs as root:root.
    sudo chown <NC_UID>:<NC_GID> ~/docker/mcpo/memory before compose up.

12. npm_config_cache=/tmp/npm_cache in mcpo environment.
    npx running as a non-root UID fails with EACCES on the default cache path.

13. ls -lnd for Nextcloud UID:GID — -d is mandatory.
    Without -d you get the owner of files inside the directory, not the
    directory itself. These may differ.

14. mcpo filesystem volume: mount user files only, not the full data dir.
    /var/srv/nextcloud/data/<username>/files:/nc-data:ro
    Full data dir includes system folders and potentially encrypted blobs.

15. HA configuration.yaml trusted_proxies required before remote access.
    Without it: 400 Bad Request from NPM proxy. Range 172.16.0.0/12 covers
    standard Docker bridges; 192.168.0.0/16 covers non-standard allocations.

16. Validate configuration.yaml before restarting HA.
    docker exec homeassistant python -m homeassistant --script check_config
    A YAML indentation error prevents HA from booting.

17. NPM WebSockets Support must be ON for both proxy hosts.
    Open WebUI streaming and HA event bus both require WebSockets.

18. HA network_mode: host — no ports: block.
    A ports: block with host networking causes port conflict errors.

19. HA requires privileged: true and /run/dbus:/run/dbus:ro per official docs.
    Omitting these causes reduced hardware access.

20. HA add-ons do not exist in Container mode.
    Do not follow tutorials referencing Supervisor, Add-on Store, or add-ons.

21. Watchtower disabled on HA.
    com.centurylinklabs.watchtower.enable: "false" prevents mid-session restarts.

22. NPM forward targets use the server's LAN IP.
    Cross-network container hostname resolution does not work between
    nextcloud-aio-network and ai-net.
