ROCM + 780M — DEFINITIVE ASSESSMENT
════════════════════════════════════════════════════════════════════

The 780M (gfx1103 / RDNA3) is usable for GPU inference on Bazzite (Fedora 42).
Three configuration requirements and one kernel parameter are mandatory.

Required environment variables in the Ollama docker-compose.yml:
  - HSA_OVERRIDE_GFX_VERSION=11.0.2
  - OLLAMA_FLASH_ATTENTION=false

  HSA_OVERRIDE_GFX_VERSION=11.0.2
    Bypasses ROCm's hardware allowlist. Without it, Ollama silently falls back
    to CPU. Must be in the container's environment block in the compose file —
    setting it in the host shell has no effect on containers.

  OLLAMA_FLASH_ATTENTION=false
    RDNA3 has a numerical instability bug in Flash Attention causing output to
    degenerate into repeated tokens (e.g., "444444..."). Disabling it eliminates
    the issue with negligible performance impact.

Required kernel parameter — mandatory before any deployment:
  sudo rpm-ostree kargs --append=amdgpu.noretry=0
  systemctl reboot

  Without amdgpu.noretry=0, the iGPU can suffer an unrecoverable MES
  (Micro Engine Scheduler) hang during sustained inference. Do not proceed
  without confirming this parameter is active.

Expected performance:
  CPU-only:    ~3–6 tok/s (7B Q4)
  780M GPU:    ~15–20 tok/s (7B Q4) · ~10–14 tok/s (14B Q4)
  Effect:      Voice end-to-end latency: ~30–40s (CPU) → ~8–12s (GPU)

BIOS prerequisite:
  The 780M uses system RAM as VRAM. A 14B Q4 model needs ~9 GB.
  BIOS UMA Frame Buffer Size must be "Auto" or ≥8 GB. GMKtec default: Auto.

eGPU note:
  When an OCuLink eGPU (RX 6000/7000 series) is added, Ollama auto-selects
  it with no configuration changes. Use ROCR_VISIBLE_DEVICES if device
  selection must be made explicit.

DOCKER NETWORK TOPOLOGY
───────────────────────

  Create once before deploying any AI stack container:
    docker network create ai-net

  This is a persistent host resource. It survives compose up/down cycles.

  Network membership:
    ai-net (bridge):  ollama, openwebui, mcpo
    host network:     homeassistant
    own default:      wyoming-whisper, wyoming-piper (ports published to host)

  Why HA is not on ai-net:
    network_mode: host bypasses all Docker bridge networks. HA cannot join
    ai-net. It reaches other services via 127.0.0.1 (loopback).

  Hostname resolution matrix:
    openwebui  → ollama:11434       via ai-net (internal Docker hostname)
    openwebui  → mcpo:8000          via ai-net (internal Docker hostname)
    homeassistant → 127.0.0.1:11434   loopback (Ollama published port)
    homeassistant → 127.0.0.1:10300   loopback (wyoming-whisper)
    homeassistant → 127.0.0.1:10200   loopback (wyoming-piper)
    NPM → <LAN-IP>:3000             NPM is on nextcloud-aio-network; cross-network
    NPM → <LAN-IP>:8123             hostname resolution does not work — use LAN IP.

  Why Ollama and Wyoming ports bind to 127.0.0.1:
    Default Docker binding (0.0.0.0) would expose the unauthenticated Ollama
    API and voice endpoints to all LAN devices. Since HA is on host network and
    shares the loopback interface, binding to 127.0.0.1 gives HA full access
    while denying access to all other LAN devices. Open WebUI connects via the
    ai-net bridge (http://ollama:11434) and is unaffected.

OLLAMA (ollama/ollama:rocm)
───────────────────────────
The :rocm tag is mandatory. The untagged ollama/ollama image is CPU-only
and ignores all GPU environment variables regardless of configuration.

  services:
    ollama:
      image: ollama/ollama:rocm
      container_name: ollama
      restart: unless-stopped
      ports:
        - "127.0.0.1:11434:11434"
      devices:
        - /dev/kfd:/dev/kfd
        - /dev/dri:/dev/dri
      group_add:
        - "<VIDEO_GID>"         # numeric GID from EX.0.d — e.g., "44"
        - "<RENDER_GID>"        # numeric GID from EX.0.d — e.g., "107"
      environment:
        - HSA_OVERRIDE_GFX_VERSION=11.0.2
        - OLLAMA_FLASH_ATTENTION=false
        - OLLAMA_KEEP_ALIVE=5m
        - OLLAMA_NUM_PARALLEL=1
        - OLLAMA_HOST=0.0.0.0
      volumes:
        - ./models:/root/.ollama
      networks:
        - ai-net

  networks:
    ai-net:
      external: true

  Why numeric GIDs:
    Docker resolves string group names from the container's own /etc/group, not
    the host's. If the group is absent from the container image, startup crashes
    with "Unable to find group render". Numeric GIDs from the host always resolve
    correctly. Obtain with: getent group video && getent group render

  Recommended model: qwen2.5:14b-instruct-q4_K_M
    ~9 GB VRAM, ~10–14 tok/s on 780M. Significantly better than 7B on multi-step
    queries, structured tool use, and long context.
    Fallback: mistral:7b-instruct-q4_K_M (~5 GB) if RAM pressure is observed.

  GPU verification:
    docker exec ollama ollama ps
    → PROCESSOR must show "100% GPU" during inference.
    If CPU: check docker inspect ollama | grep -A1 HSA — env var must be present.

MCPO (ghcr.io/open-webui/mcpo:main)
────────────────────────────────────
mcpo requires an explicit startup command pointing to the config file.
Without it, the container starts with no MCP servers loaded and exposes
no tools. The command must be declared in docker-compose.yml.

  config.json structure:
    {
      "mcpServers": {
        "filesystem": {
          "command": "npx",
          "args": ["-y", "@modelcontextprotocol/server-filesystem", "/nc-data"]
        },
        "memory": {
          "command": "npx",
          "args": ["-y", "@modelcontextprotocol/server-memory"],
          "env": {
            "MEMORY_FILE_PATH": "/app/memory/graph.json"
          }
        }
      }
    }

  Note on memory persistence:
    @modelcontextprotocol/server-memory uses the MEMORY_FILE_PATH environment
    variable to set the persistence path. This is an env field inside config.json,
    not a CLI argument. Without it, memory is lost on container restart.

  docker-compose.yml:
    services:
      mcpo:
        image: ghcr.io/open-webui/mcpo:main
        container_name: mcpo
        restart: unless-stopped
        command: --config /app/config/config.json
        user: "<NC_UID>:<NC_GID>"        # from EX.0.g — e.g., "33:33"
        environment:
          - npm_config_cache=/tmp/npm_cache
        volumes:
          - ./config.json:/app/config/config.json:ro
          - /var/srv/nextcloud/data:/nc-data:ro
          - ./memory:/app/memory
        networks:
          - ai-net

    networks:
      ai-net:
        external: true

  Why npm_config_cache=/tmp/npm_cache:
    mcpo runs as the Nextcloud UID (e.g., 33). When it executes "npx -y" to
    download MCP server packages, npm attempts to write its cache to ~/.npm
    relative to that user's home directory. For UID 33 (www-data), the home
    directory may not exist or may not be writable inside the container,
    causing EACCES (Permission Denied) and crashing the tool servers.
    Redirecting the cache to /tmp/npm_cache (which is always writable) fixes
    this. No pre-creation is needed — npm auto-creates it inside /tmp.

  Pre-deployment requirement (CRITICAL):
    The ./memory directory must exist and be owned by NC_UID:NC_GID before
    docker compose up. If Docker auto-creates it, it becomes root:root and
    mcpo will fail to write the memory graph on startup.

    mkdir -p ~/docker/mcpo/memory
    sudo chown <NC_UID>:<NC_GID> ~/docker/mcpo/memory

  Nextcloud data permissions:
    /var/srv/nextcloud/data is owned by the Nextcloud AIO internal user.
    Setting user: "<NC_UID>:<NC_GID>" ensures mcpo reads files with the
    correct identity. Use ls -lnd (not ls -la) to get numeric IDs:
      ls -lnd /var/srv/nextcloud/data
    The -n flag forces numeric output; the -d flag lists the directory itself
    rather than its contents. Without -d, you get the owner of files inside
    the directory, which may differ from the directory owner.

  Connecting Open WebUI to mcpo — per-server registration:
    When mcpo uses a config file with multiple servers, each server is
    mounted at its own URL path: http://mcpo:8000/filesystem and
    http://mcpo:8000/memory. Adding the root URL (http://mcpo:8000) results
    in no tools being discovered — the root path carries no tool definitions.
    Each server must be registered separately.

    In Open WebUI: Admin Settings → Tools → Add Tool Server (twice):
      URL: http://mcpo:8000/filesystem   Type: OpenAPI   Auth: None
      URL: http://mcpo:8000/memory       Type: OpenAPI   Auth: None

    Open WebUI auto-fetches the OpenAPI schema from each URL. Enable both
    tools globally in Admin Settings and per-model in the model settings.

  Filesystem server — what works:
    Listing directories, reading plain text files (.txt, .md, .py, .json,
    .yaml, config files).

  Filesystem server — what does NOT work:
    EPUB files (.epub) are ZIP archives. server-filesystem returns raw binary
    data — it does not parse compressed formats. Use a dedicated eBook-mcp
    server as a third entry in config.json if EPUB access is needed.

  Memory usage:
    "Remember that my ID number is X" → stored in graph.json.
    "What is my ID number?" → retrieved across sessions.
    "Forget my ID number" → deleted from the graph.


HOME ASSISTANT (ghcr.io/home-assistant/home-assistant:stable)
──────────────────────────────────────────────────────────────
Official HA-maintained image. No PUID/PGID. Requires privileged mode and
dbus access as documented at home-assistant.io/installation/linux.

  services:
    homeassistant:
      image: ghcr.io/home-assistant/home-assistant:stable
      container_name: homeassistant
      restart: unless-stopped
      privileged: true
      network_mode: host
      environment:
        - TZ=Europe/Berlin          # adjust to your timezone (TZ database format)
      volumes:
        - ./config:/config
        - /etc/localtime:/etc/localtime:ro
        - /run/dbus:/run/dbus:ro
      labels:
        com.centurylinklabs.watchtower.enable: "false"

  privileged: true is required by the official HA documentation for full
  hardware access (Bluetooth, USB devices for future Zigbee/Z-Wave).

  network_mode: host is required for mDNS/UPnP device discovery. No ports:
  block — host networking makes it redundant.

  HA add-ons do not exist. The Supervisor does not exist. Any tutorial
  referencing "install the Whisper add-on" does not apply. All integrations
  use standalone Docker containers connected via HA's native integrations.

  Reverse proxy configuration (required before remote access):
    Since HA 2021.7, reverse proxy access is blocked by default. Accessing
    home.*.duckdns.org through NPM will return "400 Bad Request" unless HA
    is configured to trust the proxy. This must be done after HA first boots
    and creates ./config/, but before testing external access.

    Edit ~/docker/homeassistant/config/configuration.yaml:

      http:
        use_x_forwarded_for: true
        trusted_proxies:
          - 172.16.0.0/12    # covers all Docker bridge networks (172.x.x.x)
          - 127.0.0.1        # loopback, for local access patterns

    Why 172.16.0.0/12:
      NPM is a Docker container on nextcloud-aio-network. When it forwards
      requests to HA via the server's LAN IP, the TCP source address HA sees
      is the NPM container's Docker bridge IP, which falls in the 172.x.x.x
      range (Docker's default bridge allocation). Using 192.168.0.0/16 (the
      LAN subnet) would not match this source and the 400 would persist.

    After editing configuration.yaml, restart HA:
      cd ~/docker/homeassistant && docker compose restart

  NPM proxy forward target: http://<LAN-IP>:8123
    NPM cannot resolve a host-networked container by hostname across separate
    Docker networks. Always use the server's LAN IP as the forward target.


VOICE PIPELINE: WYOMING-WHISPER + WYOMING-PIPER (~/docker/wyoming)
────────────────────────────────────────────────────────────────────
Both services in one docker-compose.yml. Ports bound to 127.0.0.1.
HA connects to both via 127.0.0.1 (loopback, host network).

  services:
    wyoming-whisper:
      image: rhasspy/wyoming-whisper:latest
      container_name: wyoming-whisper
      restart: unless-stopped
      ports:
        - "127.0.0.1:10300:10300"
      volumes:
        - ./whisper-data:/data
      command: --model base --language en

    wyoming-piper:
      image: rhasspy/wyoming-piper:latest
      container_name: wyoming-piper
      restart: unless-stopped
      ports:
        - "127.0.0.1:10200:10200"
      volumes:
        - ./piper-data:/data
      command: --voice en_US-lessac-high

  Model data persistence: both images use /data as the default model storage
  path. The volume mounts ./whisper-data:/data and ./piper-data:/data are
  sufficient — no additional flags are needed.

  In HA: Settings → Devices & Services → Wyoming Protocol.
  Add two entries using 127.0.0.1 (HA is on host network; loopback is correct):
    127.0.0.1 : 10300  →  Whisper STT
    127.0.0.1 : 10200  →  Piper TTS

EX.0  Pre-flight (~20 min)
  All values gathered here feed directly into compose files. Record them.

  a. Verify existing containers healthy:
       docker ps --format "table {{.Names}}\t{{.Status}}"
       → duckdns, npm, nextcloud-aio-mastercontainer: all "Up"

  b. Confirm kernel parameter (see Phase 0).

  c. Verify GPU device nodes exist:
       ls -la /dev/kfd /dev/dri/renderD128

  d. Retrieve numeric GIDs for Ollama group_add. Record both numbers:
       getent group video    → third field is the GID (e.g., 44)
       getent group render   → third field is the GID (e.g., 107)

  e. Verify available RAM:
       free -h | awk '/^Mem:/{print $7}'
       "Available" includes reclaimable cache. Aim for ≥18 GB.

  f. Verify BIOS UMA Frame Buffer Size:
       Boot into BIOS → APU/Advanced → UMA Frame Buffer Size
       Must be "Auto" or ≥8 GB (GMKtec default: Auto).

  g. Retrieve Nextcloud data owner UID:GID for mcpo. Record both numbers:
       ls -lnd /var/srv/nextcloud/data
       The -n flag forces numeric output. The -d flag lists the directory
       itself, not its contents. Note the third (UID) and fourth (GID) fields.

  h. Create shared Docker network:
       docker network create ai-net

CRITICAL PITFALLS
════════════════════════════════════════════════════════════════════

1.  Kernel parameter before everything.
    rpm-ostree kargs + reboot first. Without it, iGPU hangs the OS under
    sustained inference load. Non-negotiable first step.

2.  Wrong Ollama image tag.
    ollama/ollama:rocm is the GPU image. The default tag is CPU-only.
    GPU env vars have no effect with the wrong tag.

3.  HSA_OVERRIDE_GFX_VERSION must be in the compose environment block.
    Shell-level env vars do not propagate into containers.

4.  OLLAMA_FLASH_ATTENTION=false is mandatory.
    RDNA3 Flash Attention bug causes output to collapse to repeated tokens.

5.  Numeric GIDs in group_add, not string names.
    String names are resolved from the container's /etc/group, not the host's.
    If absent from the container, startup crashes with "Unable to find group".

6.  ai-net must exist before docker compose up on any stack that references it.
    Run docker network create ai-net before all other steps.

7.  Ollama and Wyoming ports bound to 127.0.0.1.
    HA connects to Ollama via http://127.0.0.1:11434 — not the container
    hostname (HA is not on ai-net) and not the LAN IP (no need to leave the
    loopback). Same applies for Wyoming on 10300 and 10200.

8.  mcpo requires an explicit startup command.
    command: --config /app/config/config.json must be present. Without it
    the container starts with no tools loaded.

9.  mcpo memory must use MEMORY_FILE_PATH env in config.json, not a CLI arg.
    The memory server reads the path from its environment, not from argv.

10. mcpo tools must be registered individually in Open WebUI.
    http://mcpo:8000/filesystem and http://mcpo:8000/memory — not the root.
    Adding http://mcpo:8000 discovers no tools.

11. mcpo memory directory pre-chown before docker compose up.
    Docker auto-creates missing bind-mount dirs as root:root.
    sudo chown <NC_UID>:<NC_GID> ~/docker/mcpo/memory before compose up.

12. npm_config_cache=/tmp/npm_cache in mcpo environment.
    mcpo runs as a non-root UID. npx will fail with EACCES on the default
    npm cache path if not redirected to /tmp.

13. ls -lnd for Nextcloud UID:GID — the -d flag is mandatory.
    ls -ln without -d lists directory contents, not the directory itself.
    Owner of files inside /var/srv/nextcloud/data may differ from the
    directory owner that mcpo needs.

14. HA configuration.yaml trusted_proxies required for reverse proxy access.
    Without it, https://home.*.duckdns.org returns 400 Bad Request.
    Range 172.16.0.0/12 covers Docker bridge source IPs (172.x.x.x).

15. NPM WebSockets Support must be ON for both proxy hosts.
    Without it, Open WebUI streaming breaks and HA dashboard fails to load.

16. HA network_mode: host — no ports: block.
    Adding a ports: block causes port conflict errors on startup.

17. HA uses official image (ghcr.io/home-assistant/home-assistant:stable).
    Requires privileged: true and /run/dbus:/run/dbus:ro. Omitting these
    reduces hardware access and causes Bluetooth/USB device failures.

18. HA add-ons do not exist. Do not follow any tutorial referencing
    the Supervisor, Add-on Store, or named add-ons.

19. Watchtower disabled on HA container.
    com.centurylinklabs.watchtower.enable: "false" prevents unexpected
    mid-session restarts.

20. NPM forward targets use the server's LAN IP.
    NPM is on nextcloud-aio-network. AI services are on ai-net or host network.
    Cross-network container hostname resolution does not work.
