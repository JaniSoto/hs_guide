ROCM + 780M — DEFINITIVE ASSESSMENT
════════════════════════════════════════════════════════════════════

The 780M (gfx1103 / RDNA3) IS usable for GPU inference on Bazzite (Fedora 42).
Three configuration requirements and one kernel parameter are mandatory.

Required environment variables in the Ollama docker-compose.yml:

    environment:
      - HSA_OVERRIDE_GFX_VERSION=11.0.2
      - OLLAMA_FLASH_ATTENTION=false

  HSA_OVERRIDE_GFX_VERSION=11.0.2
    Bypasses ROCm's hardware allowlist. Without it, Ollama silently falls
    back to CPU. Must be inside the container's environment block in the
    compose file — setting it in the host shell has no effect on containers.

  OLLAMA_FLASH_ATTENTION=false
    RDNA3 has a numerical instability bug in Flash Attention that causes
    output to degenerate into repeated tokens (e.g., "444444..."). Disabling
    it eliminates the issue with negligible performance impact.

Required kernel parameter — mandatory before any deployment:

    sudo rpm-ostree kargs --append=amdgpu.noretry=0
    systemctl reboot

  Without amdgpu.noretry=0, the iGPU can suffer an unrecoverable MES
  (Micro Engine Scheduler) hang during sustained inference. This is a
  hard requirement — do not proceed without it.

Expected performance on this hardware:
  CPU-only:    ~3–6 tok/s for a 7B Q4 model
  780M GPU:    ~15–20 tok/s for 7B Q4, ~10–14 tok/s for 14B Q4
  Effect:      End-to-end voice latency drops from ~30–40s to ~8–12s.

BIOS prerequisite:
  The 780M uses system RAM as VRAM. A 14B Q4 model needs ~9 GB.
  BIOS UMA Frame Buffer Size must be "Auto" or ≥8 GB.
  GMKtec default is Auto — verify in BIOS before deployment.

eGPU note:
  When an OCuLink eGPU (RX 6000/7000 series) is added, Ollama auto-selects
  it with no config changes. Use ROCR_VISIBLE_DEVICES if device selection
  must be explicit. No stack changes required at that time.

DOCKER NETWORK TOPOLOGY
───────────────────────

  Create once on the host before deploying any AI stack container:

    docker network create ai-net

  This is a persistent host resource. It survives compose up/down cycles.
  Never recreated unless manually deleted.

  Network membership:
    ai-net (bridge):  ollama, openwebui, mcpo
    host network:     homeassistant
    own default:      wyoming-whisper, wyoming-piper (ports published to host)

  Why HA is not on ai-net:
    network_mode: host bypasses all Docker bridge networks. HA cannot join
    ai-net. It reaches other services via the host's loopback or LAN IP.

  Hostname resolution matrix:
    openwebui → ollama:11434           via ai-net (Docker hostname, internal)
    openwebui → mcpo:8000              via ai-net (Docker hostname, internal)
    homeassistant → 127.0.0.1:11434   via loopback (published port, host-bound)
    homeassistant → 127.0.0.1:10300   via loopback (wyoming-whisper, host-bound)
    homeassistant → 127.0.0.1:10200   via loopback (wyoming-piper, host-bound)
    NPM → <LAN-IP>:3000               NPM is on nextcloud-aio-network; it cannot
    NPM → <LAN-IP>:8123               resolve ai-net hostnames — use LAN IP.

  Why Ollama and Wyoming ports bind to 127.0.0.1:
    Default Docker port binding (0.0.0.0) exposes the Ollama API and
    unauthenticated voice endpoints to all devices on the LAN. Since HA
    is on the host network and shares the loopback interface, binding to
    127.0.0.1 gives HA full access while denying access to all other
    LAN devices. Open WebUI is unaffected — it connects via the ai-net
    bridge (http://ollama:11434), not through the published host port.

OLLAMA (ollama/ollama:rocm)
  The :rocm tag is mandatory. The untagged ollama/ollama image is CPU-only
  and ignores GPU env vars regardless of what is configured.

  Full docker-compose.yml structure:

    services:
      ollama:
        image: ollama/ollama:rocm
        container_name: ollama
        restart: unless-stopped
        ports:
          - "127.0.0.1:11434:11434"
        devices:
          - /dev/kfd:/dev/kfd
          - /dev/dri:/dev/dri
        group_add:
          - "<VIDEO_GID>"        # numeric — obtained in EX.0.d
          - "<RENDER_GID>"       # numeric — obtained in EX.0.d
        environment:
          - HSA_OVERRIDE_GFX_VERSION=11.0.2
          - OLLAMA_FLASH_ATTENTION=false
          - OLLAMA_KEEP_ALIVE=5m
          - OLLAMA_NUM_PARALLEL=1
          - OLLAMA_HOST=0.0.0.0
        volumes:
          - ./models:/root/.ollama
        networks:
          - ai-net

    networks:
      ai-net:
        external: true

  Why numeric GIDs in group_add:
    Docker resolves string names (e.g., "video") from the container's
    own /etc/group file, not the host's. If the group is absent from
    the container image, startup fails with "Unable to find group render".
    Numeric GIDs from the host are always resolved correctly.
    Obtain with: getent group video && getent group render

  Recommended model: qwen2.5:14b-instruct-q4_K_M
    ~9 GB VRAM, ~10–14 tok/s on 780M. Significantly better than 7B on
    complex multi-step queries, structured tool use, and long context.
    Fallback: mistral:7b-instruct-q4_K_M (~5 GB) if RAM pressure is
    observed after all services are loaded.

  GPU verification command:
    docker exec ollama ollama ps
    → PROCESSOR column must show "100% GPU" during inference.
    If it shows CPU: confirm HSA_OVERRIDE_GFX_VERSION is present in
    docker inspect ollama under "Env".

MCPO (ghcr.io/open-webui/mcpo:main)
  Bridges MCP servers to Open WebUI's tool calling system. Runs two MCP
  servers as child processes: filesystem (read-only Nextcloud) and memory
  (persistent knowledge graph).

  config.json structure:

    {
      "mcpServers": {
        "filesystem": {
          "command": "npx",
          "args": ["-y", "@modelcontextprotocol/server-filesystem", "/nc-data"]
        },
        "memory": {
          "command": "npx",
          "args": ["-y", "@modelcontextprotocol/server-memory",
                   "--path", "/app/memory/graph.json"]
        }
      }
    }

  The memory server uses a knowledge graph stored in graph.json. Persistent
  across restarts because ./memory is a mounted host volume.

  docker-compose.yml:

    services:
      mcpo:
        image: ghcr.io/open-webui/mcpo:main
        container_name: mcpo
        restart: unless-stopped
        user: "<NC_UID>:<NC_GID>"    # from EX.0.g — Nextcloud data owner
        volumes:
          - ./config.json:/app/config/config.json:ro
          - /var/srv/nextcloud/data:/nc-data:ro
          - ./memory:/app/memory
        networks:
          - ai-net

    networks:
      ai-net:
        external: true

  Pre-deployment requirement (CRITICAL):
    The ./memory directory must exist and be owned by the same UID:GID
    before docker compose up. If Docker auto-creates it, it becomes
    root:root and mcpo will crash with Permission Denied on startup.

    mkdir -p ~/docker/mcpo/memory
    sudo chown <NC_UID>:<NC_GID> ~/docker/mcpo/memory

  Nextcloud data permissions:
    /var/srv/nextcloud/data is owned by the Nextcloud AIO internal user.
    Setting user: "<NC_UID>:<NC_GID>" ensures mcpo reads files with the
    correct identity. Obtain the numeric UID/GID with:
      ls -ln /var/srv/nextcloud/data
    (the -n flag forces numeric output; -l alone may return string names
    if the host resolves the UID, which is unusable in the compose user: field)

  Filesystem server — what works:
    Listing directories, reading plain text files (.txt, .md, .py, .json,
    .yaml, config files). The AI browses the Nextcloud directory tree and
    retrieves readable file content on request.

  Filesystem server — what does NOT work:
    EPUB files (.epub) are ZIP archives. server-filesystem returns raw
    binary — it does not parse compressed formats. Use the community
    eBook-mcp server (ghcr.io/alex-nt/ebook-mcp or equivalent) as a
    third entry in config.json if EPUB/book access is needed.

  Memory usage:
    "Remember that my ID number is X" → stored in graph.json.
    "What is my ID number?" → retrieved across sessions.
    "Forget my ID number" → deleted.

HOME ASSISTANT (ghcr.io/home-assistant/home-assistant:stable)
  Official HA-maintained image. No PUID/PGID needed. Requires privileged
  mode and dbus access as documented on home-assistant.io.

    services:
      homeassistant:
        image: ghcr.io/home-assistant/home-assistant:stable
        container_name: homeassistant
        restart: unless-stopped
        privileged: true
        network_mode: host
        environment:
          - TZ=<your/timezone>
        volumes:
          - ./config:/config
          - /etc/localtime:/etc/localtime:ro
          - /run/dbus:/run/dbus:ro
        labels:
          com.centurylinklabs.watchtower.enable: "false"

  privileged: true is required by the official HA documentation for full
  hardware access (Bluetooth, USB devices for future Zigbee). It is not
  optional.

  network_mode: host is required for mDNS/UPnP device discovery. There
  must be no ports: block — host networking makes it redundant. With
  network_mode: host, HA accesses the loopback at 127.0.0.1.

  HA add-ons do not exist. The Supervisor does not exist. Any tutorial
  referencing "install the Whisper add-on" does not apply here. All
  integrations use standalone Docker containers.

  NPM proxy forward target for HA: http://<LAN-IP>:8123
  Use the server's LAN IP — NPM cannot resolve a host-networked container
  by hostname across separate Docker networks.

VOICE PIPELINE: WYOMING-WHISPER + WYOMING-PIPER (~/docker/wyoming)
  Both services in one docker-compose.yml. Ports bound to 127.0.0.1.

    services:
      wyoming-whisper:
        image: rhasspy/wyoming-whisper:latest
        container_name: wyoming-whisper
        restart: unless-stopped
        ports:
          - "127.0.0.1:10300:10300"
        volumes:
          - ./whisper-data:/data
        command: --model base --language en

      wyoming-piper:
        image: rhasspy/wyoming-piper:latest
        container_name: wyoming-piper
        restart: unless-stopped
        ports:
          - "127.0.0.1:10200:10200"
        volumes:
          - ./piper-data:/data
        command: --voice en_US-lessac-high

  Model data persistence: both images use /data as the default model
  storage path. Mounting ./whisper-data:/data and ./piper-data:/data is
  sufficient for persistence. No additional flags are needed.

  network and 127.0.0.1 is the correct address):
    127.0.0.1 : 10300  → Whisper STT

IMPLEMENTATION PHASES
════════════════════════════════════════════════════════════════════

PHASE 0 — KERNEL PARAMETER (mandatory pre-condition, requires reboot)

  Execute:
    sudo rpm-ostree kargs --append=amdgpu.noretry=0
    systemctl reboot

  Verify after reboot:
    cat /proc/cmdline | grep amdgpu.noretry
    → must return "amdgpu.noretry=0"

  Do not proceed to EX.0 until this is confirmed.

EX.0  Pre-flight (~20 min)
  All values collected here feed directly into compose files. Record them.

  a. Verify existing containers healthy:
       docker ps --format "table {{.Names}}\t{{.Status}}"
       → duckdns, npm, nextcloud-aio-mastercontainer: all "Up"

  b. Verify kernel parameter (see Phase 0 above).

  c. Verify GPU device nodes exist:
       ls -la /dev/kfd /dev/dri/renderD128

  d. Retrieve numeric GIDs for Ollama group_add:
       getent group video     → note the third field (e.g., 44)
       getent group render    → note the third field (e.g., 107)
       Record both numbers. Use them — not the strings — in the compose file.

  e. Verify available RAM:
       free -h | awk '/^Mem:/{print $7}'
       "Available" includes reclaimable cache. Aim for ≥18 GB.

  f. Verify BIOS UMA Frame Buffer Size:
       Boot into BIOS → APU/Advanced → UMA Frame Buffer Size
       Must be "Auto" or ≥8 GB (GMKtec default: Auto).

  g. Retrieve Nextcloud data owner UID:GID for mcpo:
       ls -ln /var/srv/nextcloud/data
       The -n flag forces numeric output. Note the user:group numbers
       on the data directory line (e.g., 33 33 for www-data).

  h. Create shared Docker network:
       docker network create ai-net

CRITICAL PITFALLS
════════════════════════════════════════════════════════════════════

1.  Kernel parameter before everything.
    rpm-ostree kargs + reboot required before EX.0. Without it, iGPU
    causes unrecoverable OS hangs under inference load.

2.  Wrong Ollama image tag.
    ollama/ollama:rocm is the GPU image. The default tag is CPU-only.
    GPU env vars have no effect with the wrong image.

3.  HSA_OVERRIDE_GFX_VERSION and OLLAMA_FLASH_ATTENTION must be in the
    docker-compose.yml environment block. Shell-level env vars do not
    propagate into containers.

4.  Numeric GIDs in group_add, not string names.
    String names are resolved from the container's /etc/group, not
    the host's. If absent from the container, startup crashes.

5.  ai-net must exist before any compose up that references it.
    If not created, the compose file fails immediately.

6.  Ollama ports bound to 127.0.0.1, not 0.0.0.0.
    HA connects via 127.0.0.1:11434 (loopback, host network).
    Open WebUI connects via http://ollama:11434 (ai-net bridge).
    Setting HA's Ollama integration URL to the container hostname
    ("http://ollama:11434") will fail — HA is not on ai-net.

7.  mcpo memory directory must be pre-created and chowned.
    Docker auto-creates missing bind-mount dirs as root:root.
    mcpo running as the Nextcloud UID cannot write to root:root dirs.
    Run mkdir + chown BEFORE docker compose up.

8.  ls -ln for Nextcloud data UID/GID, not ls -la.
    ls -la may return string names (e.g., "www-data") which are not
    valid in Docker Compose's user: directive. Always use ls -ln.

9.  HA network_mode: host — no ports: block.
    Adding a ports: block with host networking causes port conflict
    errors at startup.

10. HA uses official image (ghcr.io/home-assistant/home-assistant:stable)
    not LSIO. Official image requires privileged: true and
    /run/dbus:/run/dbus:ro. Omitting these causes reduced hardware
    access and potential Bluetooth/device integration failures.

11. HA add-ons do not exist. Do not follow any tutorial referencing
    the Supervisor, Add-on Store, or named add-ons.

12. Watchtower disabled on HA container.
    com.centurylinklabs.watchtower.enable: "false" prevents unexpected
    mid-session restarts.

13. Open WebUI internal port is 8080. Host mapping is 3000:8080.
    Future Zigbee2MQTT also maps to host 8282:8080, not 8080:8080.

14. Wyoming ports bound to 127.0.0.1.
    HA (host network) connects via 127.0.0.1:10300 and :10200.
    Configure Wyoming integrations in HA using 127.0.0.1, not LAN IP.

15. NPM forward targets use LAN IP.
    NPM is on nextcloud-aio-network. AI services are on ai-net or host.
    Cross-network container name resolution does not work.
    Always use http://<LAN-IP>:<port> as the NPM forward target.
