ROCm / IGPU — REVISED ASSESSMENT
──────────────────────────────────

Previous assessment was wrong. The 780M IS usable for GPU inference on this
specific hardware and OS combination, with caveats.

Why it works here:
  Bazzite is based on Fedora 42, which ships ROCm packages built with gfx1103
  (780M) support. The driver stack is already present on the system.
  Ollama can use the 780M but requires one environment variable to be set:
    HSA_OVERRIDE_GFX_VERSION=11.0.2
  Without it, Ollama ignores the iGPU and falls back to CPU silently.

Expected performance gain:
  CPU-only (8845HS, 8 threads): ~3–6 tokens/sec for a 7B Q4 model.
  With 780M GPU offload:        ~15–20 tokens/sec for the same model.
  Roughly a 4–5x speedup. Voice response perceived latency drops from ~30s
  to roughly 8–12s. This is the difference between "tolerable" and "natural".

Why DDR5 4800 specifically helps:
  The iGPU has no dedicated VRAM — it uses system RAM as its memory pool.
  DDR5 4800 has approximately 30% higher bandwidth than DDR4 3200. More
  memory bandwidth directly translates to faster matrix multiply, which is
  the bottleneck for LLM inference. This is the best-case scenario for an
  iGPU-based inference setup.

Known issues that must be mitigated:
  1. MES hang / GPU reset: the iGPU can lock up without a kernel parameter.
     Fix: add amdgpu.noretry=0 to kernel boot arguments (persistent via
     rpm-ostree kargs). Without this, long inference sessions may crash.
  2. Flash Attention numerical instability: RDNA3 has a known bug in Flash
     Attention that causes model output to collapse into repeated tokens.
     Fix: set OLLAMA_FLASH_ATTENTION=false in the Ollama container environment.
  3. The HSA_OVERRIDE_GFX_VERSION env var must be set for the Ollama container,
     not just the shell. Set it as an environment variable in docker-compose.yml.
  4. /dev/kfd and /dev/dri devices must be passed into the Ollama container.
     SELinux label on /dev/kfd may need: setsebool -P container_use_devices=1
     (already covered in the main guide's SELinux setup).

Architecture change from draft v2:
  Ollama now uses the 780M iGPU for all inference.
  Recommended model upgrades to mistral:7b-instruct-q4_K_M or
  qwen2.5:14b-instruct-q4_K_M (14B now fits comfortably with GPU inference
  at ~9GB, leaving ample RAM for other services).

eGPU forward note:
  When you add an eGPU via the OCuLink port, Ollama will auto-detect it as a
  higher-priority device and use it instead of the iGPU with no config change,
  provided the eGPU is ROCm-supported (any RX 6000 or RX 7000 series card).
  Set ROCR_VISIBLE_DEVICES to control which GPU Ollama uses if needed.
  No architectural changes to the stack are required at that point.


ESP32 BOARDS — WHAT YOU HAVE IS ENOUGH FOR NOW
───────────────────────────────────────────────

Original ESP32 (WROOM / DEVKIT): works as a voice satellite via ESPHome, but
wake word detection runs on the server (Home Assistant's openWakeWord pipeline),
not on-device. Audio streams continuously from the ESP32 to HA over LAN until
the wake word fires. Fully functional, slightly higher idle bandwidth and server
CPU usage compared to S3 nodes.

ESP32-S3 (with PSRAM, N8R8 or N16R8 variant): supports on-device microWakeWord
detection — no audio leaves the device until the wake word triggers. Better
privacy, lower server load, more reliable in noisy environments.

Decision: build the core system (EX.0–EX.7) first using existing ESP32 boards
as phone/laptop/browser clients (no voice satellite hardware needed at all for
the core). When adding voice satellites (EX.8, future), start with your existing
ESP32 boards using server-side wake word detection. Upgrade specific rooms to
ESP32-S3 nodes later if on-device detection proves necessary. The ESPHome config
difference is a single flag — no rewiring required.

OLLAMA (with ROCm GPU offload)
  Image: ollama/ollama:rocm (not the standard CPU image).
  Additional docker-compose.yml settings vs draft v2:
    devices:
      - /dev/kfd
      - /dev/dri
    environment:
      HSA_OVERRIDE_GFX_VERSION: "11.0.2"
      OLLAMA_FLASH_ATTENTION: "false"
      OLLAMA_KEEP_ALIVE: "5m"
      OLLAMA_NUM_PARALLEL: "1"
      OLLAMA_HOST: "0.0.0.0"
    group_add:
      - video    (GID for /dev/dri access)
      - render   (GID for /dev/kfd access)

  Kernel boot argument (set once, persistent):
    sudo rpm-ostree kargs --append=amdgpu.noretry=0
    (prevents iGPU MES hang / GPU reset during long inference sessions)

  Recommended model: qwen2.5:14b-instruct-q4_K_M
    ~9 GB, 14B parameters. Significantly better than 7B for complex queries,
    structured tool use, and multi-step reasoning. Fits entirely in the 780M's
    accessible memory pool. ~15–18 tokens/sec expected.
    Fallback: mistral:7b-instruct-q4_K_M (~5 GB) if RAM pressure becomes an
    issue when all other services are loaded.

  Verify GPU is being used after deployment:
    docker exec ollama ollama ps
    Should show: PROCESSOR = 100% GPU (not CPU)
    Also check: docker exec ollama rocm-smi (shows GPU utilisation during query)

OPEN WEBUI + mcpo
  Unchanged from draft v2. Two MCP servers:

SMART HOME DEVICE COMPATIBILITY
  ESPHome / voice satellite nodes are entirely independent of smart home devices.
  Smart home devices are managed by Home Assistant directly via integrations.
  Any device HA supports (Zigbee, Z-Wave, Matter, Wi-Fi devices, etc.) works
  regardless of what voice satellites you use or which wake word is configured.
  Existing smart market devices: if they are Wi-Fi based and have a HA
  integration (Tuya, TP-Link Tapo, Shelly, Philips Hue, IKEA TRÅDFRI, etc.),
  they can be added to HA without any additional hardware. Zigbee devices
  require a USB coordinator dongle (EX.9).

CRITICAL PITFALLS
════════════════════════════════════════════════════════════════════

1.  amdgpu.noretry=0 kernel parameter must be set BEFORE deploying Ollama.
    Without it, the 780M iGPU may reset during long inference sessions.
    Set via: sudo rpm-ostree kargs --append=amdgpu.noretry=0 then reboot.

2.  OLLAMA_FLASH_ATTENTION=false is mandatory for gfx1103.
    With Flash Attention enabled, model output may collapse to repeated
    tokens ("444444...") due to a known RDNA3 NaN bug in Flash Attention.

3.  HSA_OVERRIDE_GFX_VERSION=11.0.2 must be set in the container environment
    in docker-compose.yml. Setting it in the shell has no effect on the container.

4.  /dev/kfd group ownership: on Bazzite, /dev/kfd may be owned by group
    "render". Confirm: ls -la /dev/kfd /dev/dri/renderD128
    Add the correct GIDs to group_add in docker-compose.yml.
    Or use devices + privileged: true as a fallback (less secure but simpler).

5.  HA network_mode: host — remove ports: block entirely.
    HA add-ons do not exist in Container mode.

6.  Watchtower must be disabled on the HA container.
    Set com.centurylinklabs.watchtower.enable: "false".

7.  Ollama :rocm image vs standard image: these are different tags.
    The standard ollama/ollama image runs CPU only.
    The GPU image is ollama/ollama:rocm. Use the correct tag.

8.  Open WebUI internal port 8080. Map to host 3000:8080.
    If Zigbee2MQTT is added later, map it to 8282:8080 — not 8080:8080.

9.  mcpo filesystem mount must be read-only (:ro).
    Never mount home directory or root filesystem as writable.

10. ESP32 WROOM boards with server-side wake word: audio streams constantly
    to HA while idle (low-level, ~16kHz mono). On a LAN this is negligible.
    If many nodes are deployed, monitor HA CPU usage from openWakeWord.
    Upgrade specific nodes to ESP32-S3 if load becomes an issue.
