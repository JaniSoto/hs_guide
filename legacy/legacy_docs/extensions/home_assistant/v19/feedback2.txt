No guide section exists for triggering the host session management scripts
(start-kde.sh, start-gaming.sh, stop-session.sh) via Home Assistant and
Wall-E. This is a desirable and architecturally clean capability: voice or
chat commands from any interface — Android app, browser, or ESP32 room node
— launch or stop server sessions without opening a terminal.

Add as a new appendix: EX.7 — HA Server Session Control (~20 min)

Prerequisites: EX.3 (Home Assistant running), base guide Phase 8 (session
scripts installed and working). Must not be placed earlier — it depends on
both the HA container and the host scripts being fully operational.

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
TECHNICAL APPROACH
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

HA runs in a container and cannot directly execute host commands. The correct
pattern is: HA shell_command integration → SSH back to the host → run the
pre-existing session script. This preserves all safety logic already built
into the scripts (session guard, cgroup-immune headless restoration via
systemd-run, DRM lock handling).

STEP 1 — Generate a dedicated SSH key for HA inside the HA container:

  docker exec -it homeassistant bash
  ssh-keygen -t ed25519 -f /config/ha_id_ed25519 -N ""
  cat /config/ha_id_ed25519.pub
  exit

STEP 2 — Authorize the key on the host:

  # On the host, append the public key printed above:
  echo "PASTE_PUBLIC_KEY_HERE" >> ~/.ssh/authorized_keys
  chmod 600 ~/.ssh/authorized_keys

STEP 3 — Pre-populate SSH known_hosts inside HA to avoid interactive prompt:

  docker exec -it homeassistant bash
  ssh-keyscan -p 22022 192.168.x.x >> /config/known_hosts
  exit

STEP 4 — Add shell_command entries to configuration.yaml:

  shell_command:
    session_kde: >
      ssh -i /config/ha_id_ed25519
          -o UserKnownHostsFile=/config/known_hosts
          -p 22022 USERNAME@192.168.x.x
          'sudo start-kde.sh'
    session_gaming: >
      ssh -i /config/ha_id_ed25519
          -o UserKnownHostsFile=/config/known_hosts
          -p 22022 USERNAME@192.168.x.x
          'sudo start-gaming.sh'
    session_stop: >
      ssh -i /config/ha_id_ed25519
          -o UserKnownHostsFile=/config/known_hosts
          -p 22022 USERNAME@192.168.x.x
          'sudo stop-session.sh'

  Replace USERNAME and 192.168.x.x with your actual values.
  Replace 22022 if your SSH port differs.

STEP 5 — Wrap each shell_command as a named HA script
(scripts expose a friendly name and description to the Ollama conversation
agent, making intent clearer than raw shell_command entity names):

  script:
    launch_kde:
      alias: "Launch KDE Desktop"
      sequence:
        - service: shell_command.session_kde

    launch_gaming:
      alias: "Launch Gaming Mode"
      sequence:
        - service: shell_command.session_gaming

    stop_session:
      alias: "Stop Active Session"
      sequence:
        - service: shell_command.session_stop

STEP 6 — Validate config and restart HA:

  docker exec homeassistant python -m homeassistant \
    --script check_config --config /config
  # → must print "Configuration is valid!"

  cd ~/docker/homeassistant && docker compose restart

STEP 7 — Enable script control in the Ollama integration:

  HA → Settings → Devices & Services → Ollama → Configure
  → "Control Home Assistant" must be ON
  → Context window: 8192

  With Control Home Assistant enabled, Wall-E has access to all HA
  entities including the scripts defined above. No additional configuration
  is required — Wall-E will discover and call them by their alias names.

FUNCTIONAL TESTS
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Test 1 — Direct HA trigger (verify shell_command works before involving AI):
  HA → Developer Tools → Services → script.launch_gaming → Call Service
  → Gaming Mode must appear on the physical display.
  → Call script.stop_session → server returns to headless idle.

Test 2 — Wall-E voice trigger:
  Say: "Wall-E, launch gaming mode"
  → Wall-E calls the HA script entity
  → Gaming Mode appears on the display (~5–10 s total)

Test 3 — Guard behaviour:
  While Gaming Mode is running, say: "launch KDE"
  → start-kde.sh refuses (session already active on seat0)
  → Wall-E reports the session is already in use

Test 4 — Stop via voice:
  Say: "stop the session" or "para la sesión"
  → stop-session.sh runs, server returns to headless idle
  → Wall-E confirms

SECURITY NOTES FOR THE GUIDE
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

- The HA SSH key must be authorized for the regular user account only
  (not root). The existing sudoers entry already grants NOPASSWD for the
  three script paths — no additional sudo configuration is needed.
- The SSH key is stored inside the HA config volume (~/docker/homeassistant/
  config/). This directory is bind-mounted with :z and already protected by
  the same access controls as the rest of the HA config.
- shell_command in HA runs as the homeassistant container user internally
  but SSH authenticates as the host user. The key must match the host user
  that has sudo rights for the session scripts.
- Only three specific commands are exposed. HA has no general shell access
  to the host. This is the correct minimal-privilege posture.
- Do not use StrictHostKeyChecking=no in the SSH command. The known_hosts
  pre-population in Step 3 is the correct approach.

Session control goes through HA entities directly — the same path as all
other home automation commands. The Modelfile already instructs Wall-E to
use HA entities directly for device control with no tool needed. The script
aliases (Launch KDE Desktop, Launch Gaming Mode, Stop Active Session) are
descriptive enough that Wall-E selects them correctly from natural language
without additional prompt engineering.
