2.1 — UMA FRAME BUFFER SIZE: CORRECT VALUE IS 8 GB (NOT "AUTO")
The guide states "Auto or ≥ 8 GB — Auto is strongly recommended."
On the GMKtec NucBox K8 BIOS, the Auto option does not exist. Range is 1-16 GB.

  WRONG:  "UMA Frame Buffer Size: must be Auto or ≥ 8 GB. Auto is strongly recommended."
  CORRECT: "UMA Frame Buffer Size: set to 8 GB."

Rationale:
- qwen2.5:7b Q4_K_M requires 4.7 GB model weights + 448 MB KV cache +
  494 MB compute buffer = ~5.7 GB total. 8 GB provides clean headroom with
  no GTT spill.
- 3 GB (factory default on this unit) causes the model to overflow into GTT
  by ~1.7 GB from load, placing the memory controller under split-pool pressure
  during inference and meaningfully increasing GPU hang risk.
- Values above 8 GB permanently remove RAM from the OS even when Ollama is
  idle, creating memory pressure for Nextcloud AIO, HA, and mcpo concurrently.
- With 32 GB DDR5, 8 GB UMA leaves 24 GB for the OS and all Docker stacks.

2.2 — POWER MODE: SET TO PERFORMANCE
The guide does not mention BIOS power mode. Add to EX.0 Pre-flight:

  BIOS → Power Mode: set to Performance (not Balanced).

Rationale: Balanced mode throttles GPU clocks during sustained inference,
causing tok/s to drop mid-generation on longer tool-call responses. Performance
mode allows the 780M to sustain its full clock speed (~2800 MHz) throughout.
LLM inference is bursty — thermal risk is minimal.

3.1 — SECOND MANDATORY KERNEL PARAMETER: amdgpu.cwsr_enable=0
The existing amdgpu.noretry=0 parameter is necessary but not sufficient.

Root cause: CWSR (Compute Wave Save/Restore) is a GPU preemption mechanism.
On gfx1103 (RDNA3 iGPU) with ROCm, when a long compute wave (such as a
tool-call inference cycle) is interrupted, the MES (Micro Engine Scheduler)
fails to cleanly remove the queue, triggering a GPU reset. The dmesg signature:

  amdgpu: MES failed to respond to msg=REMOVE_QUEUE
  amdgpu: MES might be in unrecoverable state, issue a GPU reset
  level=ERROR source=server.go msg="post predict" error="...EOF"

Symptom in Open WebUI: blank response with a white dot. Ollama logs show the
runner process crashing with SIGABRT (core dumped). Open WebUI receives EOF.

amdgpu.noretry=0 prevents permanent MES lockout on retry failure but does NOT
prevent CWSR preemption from firing. amdgpu.cwsr_enable=0 disables compute
wave preemption entirely, eliminating the trigger.

Why tool calls specifically trigger it: tool-call cycles sustain GPU compute
for 2-3x longer than plain generation (system prompt + tool definitions + user
message + tool call JSON + tool result + final generation). This longer
duration crosses the CWSR preemption threshold.

Confirmed fix (applies to 7840HS, 8845HS, all RDNA3 iGPU variants):
  sudo rpm-ostree kargs --append=amdgpu.cwsr_enable=0
  sudo systemctl reboot

Trade-off: some compute-heavy games may exhibit stutter. For a home server
primarily doing LLM inference, this is the correct trade-off. Fully reversible:
  rpm-ostree kargs --delete=amdgpu.cwsr_enable=0

The guide's "Required Kernel Parameter" section must list BOTH parameters:

  sudo rpm-ostree kargs --append=amdgpu.noretry=0
  sudo rpm-ostree kargs --append=amdgpu.cwsr_enable=0
  sudo systemctl reboot

  # After reboot, verify BOTH are active:
  cat /proc/cmdline | grep -o "amdgpu\.[^ ]*"
  # Must show: amdgpu.noretry=0 AND amdgpu.cwsr_enable=0

  cat /sys/module/amdgpu/parameters/cwsr_enable
  # Must output: 0. Do not proceed until it does.

The guide instructs setting Tool Calling to Native for wall-e. This is broken
due to Open WebUI issue #17135 (role:tool content:null malformed message).

Symptom: blank response with a white dot after any tool READ operation. Ollama
returns HTTP 200 successfully. The failure is in Open WebUI's second pass —
when it sends the tool result back to the model for a final answer, the message
is malformed and produces no output. Confirmed by switching to Legacy, which
resolved the blank immediately.

  WRONG:  "Tool Calling: set to Native"
  CORRECT: "Tool Calling: set to Legacy"

Legacy injects tool definitions via prompt rather than native function calling.
~1-2 seconds slower on tool-heavy requests; functionally equivalent for all
use cases in this guide.

Revert to Native when #17135 is resolved:
  https://github.com/open-webui/open-webui/issues/17135
