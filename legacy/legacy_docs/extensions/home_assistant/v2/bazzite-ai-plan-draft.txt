ROCM + 780M — DEFINITIVE ASSESSMENT
════════════════════════════════════════════════════════════════════

The 780M (gfx1103 / RDNA3) is usable for GPU inference on this specific
hardware and OS combination.

Bazzite is based on Fedora 42, which ships ROCm packages with gfx1103 support
in the kernel driver stack. The key constraint is that Ollama does not auto-
detect the 780M as a supported GPU without a manual override.

Required configuration in docker-compose.yml:
  environment:
    HSA_OVERRIDE_GFX_VERSION: "11.0.2"
    OLLAMA_FLASH_ATTENTION: "false"

  HSA_OVERRIDE_GFX_VERSION: "11.0.2"
    Bypasses ROCm's hardware allowlist. Without it, Ollama ignores the iGPU
    and silently falls back to CPU. Must be set in the container environment
    in docker-compose.yml — setting it in the host shell has no effect.

  OLLAMA_FLASH_ATTENTION: "false"
    RDNA3 has a numerical instability bug in its Flash Attention implementation
    that causes LLM output to degenerate into repeated tokens (e.g. "44444...").
    This is a known gfx1103 hardware-level issue. Disabling Flash Attention
    eliminates it entirely with negligible performance impact.

Required kernel parameter (set before any deployment):
  sudo rpm-ostree kargs --append=amdgpu.noretry=0
  systemctl reboot

  Without amdgpu.noretry=0, the iGPU can suffer a MES (Micro Engine Scheduler)
  hang during sustained inference. The hang is unrecoverable without a reboot.
  This is the single most important pre-condition for stable GPU inference.

Expected performance on this hardware:
  CPU-only:     ~3–6 tokens/sec  (7B Q4 model)
  780M GPU:     ~15–20 tokens/sec (7B Q4 model), ~10–14 tokens/sec (14B Q4)
  Speedup:      ~4–5x

  With GPU inference, voice end-to-end latency (wake word → first spoken word)
  drops from ~30–40s to ~8–12s. This is the practical threshold for a usable
  voice assistant.

BIOS prerequisite:
  The 780M has no dedicated VRAM. It dynamically allocates from system RAM.
  A 14B Q4 model requires ~9 GB. The BIOS UMA Frame Buffer Size must be set to
  "Auto" or "8GB+" — GMKtec's default is Auto, which is correct. Verify in BIOS
  before deployment if GPU usage fails to show >4GB allocation.

eGPU note:
  When an OCuLink eGPU (any RX 6000/7000 series) is added, Ollama auto-detects
  it as the primary device with no configuration change required. To force a
  specific device use ROCR_VISIBLE_DEVICES. No stack changes needed at that time.

DOCKER NETWORK TOPOLOGY
───────────────────────

  Services in separate compose directories cannot resolve each other by
  hostname unless they share an external Docker network. The following
  named network must be created before deploying any AI stack container.

  docker network create ai-net

  This is a persistent host-level resource. It survives container restarts
  and compose up/down cycles. Created once; never recreated unless deleted.

  Network membership:
    ai-net:        ollama, openwebui, mcpo
    host network:  homeassistant, (matter-hub if added later)
    own default:   wyoming-whisper, wyoming-piper (ports published to host)

  Why HA and wyoming are not on ai-net:
    Home Assistant uses network_mode: host, which bypasses all Docker bridge
    networks. It cannot join ai-net. It reaches Ollama via the host LAN IP
    on port 11434 (published by the Ollama container to the host).
    wyoming-whisper and wyoming-piper publish ports 10300 and 10200 to the
    host. HA (on host network) accesses them via <LAN-IP>:<port>.

  Hostname resolution summary:
    openwebui → ollama:11434      (via ai-net, internal hostname)
    openwebui → mcpo:8000         (via ai-net, internal hostname)
    homeassistant → <LAN-IP>:11434  (via host network, published port)
    homeassistant → <LAN-IP>:10300  (wyoming-whisper, published port)
    homeassistant → <LAN-IP>:10200  (wyoming-piper, published port)
    NPM → openwebui:3000          (NPM is on nextcloud-aio-network; needs
                                   openwebui to also join that network, or
                                   NPM can forward to <LAN-IP>:3000 instead)

  NPM proxy note:
    NPM is already running on nextcloud-aio-network. To proxy to Open WebUI
    and HA without container hostname resolution, use the server's LAN IP as
    the forward target in NPM (e.g., http://<LAN-IP>:3000 for Open WebUI,
    http://<LAN-IP>:8123 for HA). This is identical to how the main guide
    already handles Nextcloud AIO's internal containers.

COMPONENT RATIONALE

OLLAMA (ollama/ollama:rocm)
  The :rocm image tag is mandatory — the standard ollama/ollama image is CPU
  only and will ignore the GPU regardless of environment variables.

  docker-compose.yml required settings:
    image: ollama/ollama:rocm
    ports:
      - "11434:11434"          ← required: HA (host network) reaches Ollama
                                  via LAN IP:11434; container port alone is
                                  not accessible from host network.
    devices:
      - /dev/kfd               ← GPU compute device
      - /dev/dri               ← GPU render device
    group_add:
      - "<VIDEO_GID>"          ← numeric GID, not the string "video"
      - "<RENDER_GID>"         ← numeric GID, not the string "render"
    environment:
      HSA_OVERRIDE_GFX_VERSION: "11.0.2"
      OLLAMA_FLASH_ATTENTION: "false"
      OLLAMA_KEEP_ALIVE: "5m"
      OLLAMA_NUM_PARALLEL: "1"
      OLLAMA_HOST: "0.0.0.0"
    networks:
      - ai-net
    networks (top-level):
      ai-net:
        external: true

  Why numeric GIDs:
    Docker resolves string group names (e.g., "video") by looking inside the
    container's /etc/group file, not the host's. The ollama:rocm container may
    not define these groups. Using numeric GIDs from the host is deterministic.
    Obtain them before writing the compose file:
      getent group video    → e.g., video:x:44:...    → use "44"
      getent group render   → e.g., render:x:107:...  → use "107"
    (exact values vary; always read them from your system)

  Recommended model: qwen2.5:14b-instruct-q4_K_M
    ~9 GB VRAM usage. ~10–14 tokens/sec on 780M. Significantly better than
    7B on complex queries, multi-step reasoning, and structured tool use.
    Fits within the 32GB DDR5 pool with room for all other services.
    Fallback: mistral:7b-instruct-q4_K_M (~5 GB) if RAM pressure is observed
    after all services are loaded.

  Verify GPU is in use after first model pull:
    docker exec ollama ollama ps
    → PROCESSOR column must show "100% GPU"
    If it shows CPU, HSA_OVERRIDE_GFX_VERSION is not being applied to the
    container environment. Check docker inspect for the env var.

OPEN WEBUI (ghcr.io/open-webui/open-webui:main)
  Primary interface. Connects to Ollama via ai-net hostname: ollama:11434.
  Mobile-responsive. Supports voice input via browser mic permissions.
  Persistent conversation history. RBAC for multiple users.

  docker-compose.yml key settings:
    ports: - "3000:8080"
    environment:
      OLLAMA_BASE_URL: http://ollama:11434
      WEBUI_SECRET_KEY: <random 32-char string, generate once>
    volumes: - ./data:/app/backend/data
    networks: - ai-net

  File access (plain text, PDF, etc.):
    Open WebUI has built-in RAG (document upload into the chat). For
    structured file browsing from Nextcloud, use the MCP filesystem tool
    via mcpo. Plain text files (.txt, .md, .json, config files) are read
    correctly. See the mcpo section for format limitations.

MCPO (ghcr.io/open-webui/mcpo:main)
  MCP-to-OpenAPI proxy. Runs MCP servers as child processes and exposes
  them as HTTP tool endpoints that Open WebUI calls. Port 8000 (ai-net
  internal only — not published to host).

  Two MCP servers configured in mcpo's config.json:
    @modelcontextprotocol/server-filesystem  (read-only Nextcloud data path)
    @modelcontextprotocol/server-memory       (writable memory store)

  Filesystem server — what works:
    Listing directories, reading plain text files (.txt, .md, .py, .json,
    .yaml, config files, etc.), reading PDFs as raw text. The AI can browse
    the Nextcloud directory tree and retrieve readable file content.

  Filesystem server — what does NOT work:
    EPUB files. EPUB is a ZIP archive. server-filesystem returns raw binary
    data for non-text files — it does not parse ZIP-based formats into
    chapters. The statement "ask the AI to read chapter 3 of mybook.epub"
    was incorrect in earlier drafts.
    Alternative: the community eBook-mcp server
    (ghcr.io/alex-nt/ebook-mcp or equivalent) is specifically designed for
    EPUB/PDF book interaction. This can be added as a third MCP server in
    mcpo's config alongside the filesystem and memory servers.

  Nextcloud volume permission requirement:
    The Nextcloud data directory (/var/srv/nextcloud) is owned by the UID of
    the Nextcloud AIO container's data user. If mcpo runs as a different UID,
    direct volume reads will return "Permission Denied".
    Fix: check the owner UID of the data path before writing the compose file:
      ls -la /var/srv/nextcloud/data
    Then set in the mcpo compose file:
      user: "<UID>:<GID>"     ← matching the Nextcloud data owner
    Mount the path read-only:
      - /var/srv/nextcloud/data:/nc-data:ro

  Memory server:
    Writable key-value and knowledge-graph store. Write path is isolated to a
    dedicated ./memory volume inside the mcpo container. The AI cannot write
    to any other location.
    Usage: "Remember that my ID number is X" → stored. "What is my ID number?"
    → retrieved across sessions. "Forget my ID number" → deleted.

  docker-compose.yml key settings:
    ports: (none — internal only on ai-net)
    volumes:
      - ./config.json:/app/config/config.json:ro
      - /var/srv/nextcloud/data:/nc-data:ro
      - ./memory:/app/memory
    user: "<UID>:<GID>"       ← Nextcloud data owner UID/GID
    networks: - ai-net

HOME ASSISTANT CONTAINER (lscr.io/linuxserver/homeassistant:latest)
  Smart home hub. Manages all devices, runs the voice pipeline, and connects
  to Ollama as its conversation agent.

  Critical compose settings:
    network_mode: host         ← mandatory for mDNS/device discovery
    (no ports: block)          ← host networking makes it redundant
    labels:
      com.centurylinklabs.watchtower.enable: "false"

  HA Container vs HAOS:
    HA Container has no Supervisor, therefore no add-ons. Any tutorial or
    documentation referring to "install the Whisper add-on" or "install the
    Piper add-on" does not apply. Whisper and Piper run as separate Docker
    containers connected via the Wyoming Protocol integration. The result
    is functionally identical; only the installation path differs.

  NPM proxy forward target:
    Because HA is on host networking, NPM cannot resolve it by container name.
    Use the server's LAN IP: http://<LAN-IP>:8123
    This is the same pattern already used for Nextcloud AIO in the main guide.

VOICE PIPELINE: WYOMING-WHISPER + WYOMING-PIPER (~/docker/wyoming)
  Both services defined in a single docker-compose.yml in ~/docker/wyoming.

  wyoming-whisper:
    image: rhasspy/wyoming-whisper:latest
    ports: - "10300:10300"
    command: --model base --language en
    Model: base (~145 MB). Adequate for home control commands. Upgrade to
    "small" if accuracy is poor on proper nouns.

  wyoming-piper:
    image: rhasspy/wyoming-piper:latest
    ports: - "10200:10200"
    command: --voice en_US-lessac-high
    Model: en_US-lessac-high. Natural-sounding voice. ~0.5s synthesis per
    sentence on CPU. Streams to HA so first words play before full response
    is generated.

  Neither container needs to be on ai-net. HA reaches them via LAN IP on
  their published ports.

  In HA: Settings → Devices & Services → Wyoming Protocol.
  Add two entries:
    <LAN-IP>:10300  (Whisper STT)
    <LAN-IP>:10200  (Piper TTS)
  HA auto-detects and names both.

ESP32 VOICE SATELLITE NODES (future, EX.8)
  Existing ESP32 WROOM boards work. Wake word runs server-side in HA
  (openWakeWord). Audio streams from the ESP32 to HA over LAN when
  triggered. Per-node additional hardware: INMP441 I2S mic (~€3),
  MAX98357A I2S amp (~€3), 4Ω speaker (~€5). Total per node ~€11.

  Wake word "Wall-E": train with openWakeWord on the server using Piper
  TTS to generate synthetic audio. Use "hey jarvis" as a placeholder
  during development.

  Upgrade to on-device wake word: replace WROOM board with ESP32-S3 N8R8
  (~€8). All other components and wiring are identical. Change is one
  flag in the ESPHome YAML config.

  Smart home devices are entirely unaffected by the voice satellite design.
  HA manages all devices independently.

ZIGBEE (future, EX.9)
  Hardware: HA Connect ZBT-2 (~€30, Zigbee + Thread) or Sonoff ZBDongle-E
  (~€20, Zigbee only). Always use a 20–30 cm USB 2.0 extension cable.
  Stack: Mosquitto + Zigbee2MQTT in ~/docker/zigbee/. USB dongle addressed
  by /dev/serial/by-id/ path (never /dev/ttyUSB0, which changes on reboot).

SYSTEM COMMAND TOOLS — REMOVED FROM PLAN
  Earlier drafts proposed an Open WebUI Python tool for running host commands
  (df, docker ps, systemctl, etc.). This is not architecturally feasible as
  described. Docker containers are isolated from the host filesystem and process
  namespace. Inside any container: df -h shows the container's overlay
  filesystem; docker ps requires the Docker socket to be mounted (a critical
  security vulnerability); systemctl requires systemd bus access that does not
  exist inside a container.

  The same isolation problem applies to Home Assistant's shell_command
  integration — it also runs inside the HA container, not on the host.

  The correct future approach is a dedicated privileged sidecar container
  with a minimal HTTP API (a small FastAPI or Flask service) that wraps a
  strict whitelist of host commands via the Docker socket or via SSH to the
  host. This sidecar is the only process that touches privileged host
  resources; the AI calls it via a restricted API, never directly. This
  requires careful separate design and is out of scope for this plan.

IMPLEMENTATION PHASES
════════════════════════════════════════════════════════════════════

PHASE 0 — KERNEL PARAMETER (before anything else)
  This must be done before EX.0. The system must be rebooted before
  proceeding. Do not skip this.

    sudo rpm-ostree kargs --append=amdgpu.noretry=0
    systemctl reboot

  After reboot, verify:
    cat /proc/cmdline | grep amdgpu.noretry
    → must return "amdgpu.noretry=0"

  If the parameter is absent after reboot, the kargs command did not
  persist. This indicates an rpm-ostree issue. Do not proceed until
  verified.

EX.0  Pre-flight (~20 min)
  Run all checks before writing any compose file.

  a. Verify existing containers healthy:
       docker ps --format "table {{.Names}}\t{{.Status}}"
       → duckdns, npm, nextcloud-aio-mastercontainer must show "Up"

  b. Verify kernel parameter (see Phase 0 above).

  c. Verify /dev/kfd and /dev/dri/renderD128 exist:
       ls -la /dev/kfd /dev/dri/renderD128

  d. Retrieve numeric GIDs for Ollama compose file. Record these values:
       getent group video
       getent group render
       → note the third colon-separated field (the GID number)

  e. Verify available RAM:
       free -h | awk '/^Mem:/{print $7}'
       → aim for ≥18 GB free after existing containers are loaded

  f. Verify BIOS UMA Frame Buffer Size:
       Boot into BIOS → Advanced/APU settings → UMA Frame Buffer Size
       → must be "Auto" or ≥8GB. GMKtec default is Auto (correct).

  g. Retrieve Nextcloud data owner UID/GID for mcpo compose file:
       ls -la /var/srv/nextcloud/data
       → note the user:group of the data directory

  h. Create the shared AI network:
       docker network create ai-net

EX.1  Ollama with iGPU (~45 min, mostly model download time)
  mkdir -p ~/docker/ollama && cd ~/docker/ollama

  Write docker-compose.yml with:
    image: ollama/ollama:rocm                 ← not the standard image
    ports: - "11434:11434"
    devices: /dev/kfd, /dev/dri
    group_add: [<VIDEO_GID>, <RENDER_GID>]    ← numeric values from EX.0.d
    environment: HSA_OVERRIDE_GFX_VERSION, OLLAMA_FLASH_ATTENTION=false,
                 OLLAMA_KEEP_ALIVE=5m, OLLAMA_NUM_PARALLEL=1,
                 OLLAMA_HOST=0.0.0.0
    volumes: ./models:/root/.ollama
    networks: ai-net (external: true)

  cd ~/docker/ollama && docker compose up -d

  Pull model (one-time ~8.9 GB download):
    docker exec ollama ollama pull qwen2.5:14b-instruct-q4_K_M

  Verify GPU inference:
    docker exec ollama ollama ps
    → PROCESSOR must show "100% GPU"
    If it shows CPU: check docker inspect ollama | grep HSA — the env var
    must be present. If absent, the compose file env block has an error.

  Functional verification:
    curl http://localhost:11434/api/generate \
      -d '{"model":"qwen2.5:14b-instruct-q4_K_M","prompt":"Hello","stream":false}'
    → Returns JSON with a "response" field. First run loads the model (~15s).

CRITICAL PITFALLS
════════════════════════════════════════════════════════════════════

1.  Kernel parameter before all else.
    sudo rpm-ostree kargs --append=amdgpu.noretry=0 then reboot.
    Without it, GPU hangs during inference. Non-negotiable first step.

2.  Wrong Ollama image tag.
    ollama/ollama:rocm is the GPU image. ollama/ollama (no tag) is CPU only.
    The GPU env vars have no effect with the wrong image.

3.  HSA_OVERRIDE_GFX_VERSION must be in docker-compose.yml environment block.
    Setting it in the terminal before running docker compose up does not
    propagate into the container. Verify with: docker inspect ollama.

4.  OLLAMA_FLASH_ATTENTION=false is mandatory on gfx1103.
    With it enabled, output collapses to repeated tokens mid-response.

5.  Numeric GIDs in group_add.
    group_add: [video, render] (string names) will crash container init if
    those groups are not defined in the container's /etc/group.
    Always use the numeric values from: getent group video render.

6.  Ollama port 11434 must be published to the host.
    HA runs on host network and cannot reach containers via bridge hostnames.
    Without ports: - "11434:11434", HA cannot connect to Ollama.

7.  ai-net must exist before docker compose up.
    If docker network create ai-net has not been run, any compose file with
    external: true for ai-net will fail immediately at startup.

8.  HA network_mode: host — no ports: block.
    Port conflicts and startup failures result from including a ports: block
    with host networking. Remove it entirely.

9.  HA add-ons do not exist. Do not follow HA Container tutorials that
    reference the Supervisor, the Add-on Store, or any add-on by name.
    All integrations in this setup use standalone Docker containers.

10. Watchtower must be disabled on HA.
    An unexpected HA restart mid-session drops voice pipeline state.
    Set com.centurylinklabs.watchtower.enable: "false" on the HA container.

11. mcpo volume permission.
    Nextcloud data files are owned by a specific UID. Set user: in mcpo's
    compose file to the matching UID:GID (verified in EX.0.g).
    If not set and permissions are restrictive, all filesystem reads fail.

12. Open WebUI internal port is 8080. Map to host 3000:8080.
    If Zigbee2MQTT is added later, map it to 8282:8080 — not 8080:8080.

13. NPM forward target for HA and Open WebUI: use the server's LAN IP.
    NPM is on nextcloud-aio-network. Open WebUI is on ai-net. They cannot
    resolve each other by container name across separate networks.
    Use http://<LAN-IP>:3000 and http://<LAN-IP>:8123 as forward targets.

14. First Ollama query after idle: model unloads after OLLAMA_KEEP_ALIVE=5m.
    Next query after unload takes ~15s to reload. Expected behaviour.
